## 主題一：Spring Initializr 初始選型、組態架構決策與版本生命週期評估

### 【問題現象與痛點】

在專案規劃初期，若不理解 [Spring Initializr 官網](https://start.spring.io/) 上各項選型的底層技術代價，盲目勾選了帶有 `SNAPSHOT` 或 `M1` 字尾的 Spring Boot 版本、選錯了傳統的 `War` 打包格式，或是在 Maven 與 Gradle 之間舉棋不定，將大幅拉高後續全端部署的運維成本，甚至導致專案因版本不相容而難以維護。

### 【底層架構與核心原理】

* **建置工具選型演進（Maven vs Gradle）：**
* **Maven**：採用宣告式的 XML 語法（`pom.xml`），結構較為臃腫且標籤重複性高。每次構建皆需重新檢查依賴，速度較慢。
* **Gradle**：將建置邏輯視為「寫程式」。其核心具備強大的增量建置（Incremental Builds）**與**建置快取（Build Cache）機制，構建速度大幅領先。在語法上分為傳統具備極大彈性的 `Gradle - Groovy`（`build.gradle`）與現代官方推薦的靜態強型別 **`Gradle - Kotlin`（`build.gradle.kts`）**。後者因具備靜態型別約束，能讓 IDE 提供極其精準的自動補完與錯誤提示。
* **決策**：初衷：因為 MVP 初期變動極大，需要頻繁增減依賴。
  關鍵點：看中了 Gradle 的「增量建置」能力（只編譯變動的部分，節省時間）以及強大的「構建快取」（Build Cache），這能確保在本地 Windows 環境下，即使 C 槽空間不足，也能透過設定將快取重導至 D 槽，不會因為編譯太慢而打斷開發節奏。

* **Spring Boot 版本生命週期字尾定義：**
* `SNAPSHOT`（快照版）：代表該版本仍處於密集開發階段，代碼隨時可能被覆蓋，極度不穩定，絕不可用於生產環境。
* `M (Milestone)` / `RC (Release Candidate)`（里程碑版/發行候選版）：供社群測試選用，仍有潛在的架構調整與 Bug。
* **無字尾（GA - General Availability / Stable）**：官方認證的穩定正式版，經過完整測試，是長期自行維運專案的唯一選擇。


* **打包封裝格式的元件隔離（Jar vs War）：**
* `War`（Web Application Archive）：傳統 Java Web 格式，必須外掛部署於獨立安裝的外部 Servlet 容器（如 Tomcat），維護成本高。
* `Jar`（Java Archive）：內建**嵌入式 Tomcat 伺服器（Embedded Tomcat）**，最符合微服務精神。只需透過作業系統一行指令 `java -jar app.jar` 即可在任何環境執行，是現代容器化（Docker）部署的絕對核心。


* **Java 語言版本斷代與虛擬執行緒效益（Java 21 LTS）：**
* Spring Boot 3.x 以上世代強制要求最低基線為 Java 17。
* **Java 21** 引入了革命性的**虛擬執行緒（Virtual Threads）**。傳統 Java 採取「一請求一執行緒」模型，極度消耗作業系統記憶體。虛擬執行緒由 JVM 自行調度，記憶體開銷極低，能以極少資源處理海量併發請求，能大幅壓低雲端伺服器的租用成本。


### 【實戰解決方案與配置對應】

建立長期維運、前後端分離專案的官方最優初始組態配置設定：

| 欄位名稱 | 生產環境最優設定值 | 長期維運考量與架構價值 |
| --- | --- | --- |
| **Project** | Gradle - Kotlin | 享受動態精簡配置、極速快取建置與 IDE 100% 精準自動補完 |
| **Language** | Java | 最成熟穩定的後端核心生態 |
| **Spring Boot** | 最新穩定 GA 版本 (如 3.4.x / 4.0.x 穩定版) | 拒絕 SNAPSHOT，確保第三方套件完美相容 |
| **Packaging** | Jar | 內建嵌入式 Tomcat，實現無痛 Docker 容器化部署 |
| **Java** | 21 | 當前效能、省錢與相容性的完美平衡點（激活虛擬執行緒功能） |

**維運必帶的「基礎相依套件 (Dependencies)」勾選清單：**

4. **Spring Data JPA + PostgreSQL Driver**：標準 ORM 操作工具，適配雲端與本地關聯式資料庫。

## 主題二：Spring Initializr 專案中繼資料定義與 Java 套件命名規範之衝突

在初始化專案時，理解這三個欄位的本質區別，是避免後續開發期出現莫名編譯錯誤的「防呆」關鍵。

### 【三大欄位身分證定義】

| 欄位 | 用途 | 是否能用連字號 (`-`) | 代表範例 |
| --- | --- | --- | --- |
| **Group** | 識別組織 (反向網域) | 否 (用點分隔) | `com.echogallery` |
| **Artifact** | 專案名稱 (產生目錄/Jar檔名) | **是** | `echo-gallery-backend` |
| **Package Name** | 程式碼家門 (Java 語法路徑) | **否** | `com.echogallery.backend` |

1. **Group (組織識別碼)**：專案的歸屬地。命名規範採用**反向網域名稱**（如 `com.echogallery`），目的是確保在全球套件庫中唯一，未來若要打包成 Library 供他人引用時，這是你的唯一 ID。
2. **Artifact (專案識別名稱)**：專案本身的正式名稱。這是建置工具（Maven/Gradle）辨識你專案目錄的關鍵字。**它是唯一允許使用連字號 `-` 的地方**，因為它只出現在檔案系統名稱與打包後的 `.jar` 檔名，不會進入程式碼邏輯。
3. **Package Name (Java 的代碼家門)**：這是你 Java 程式碼中 `package` 宣告的路徑。它是 Java 執行環境的階層根節點。

### 【問題現象與痛點】

在定義 Project Metadata 時，若在 Artifact 欄位輸入帶有連字號（如 `echo-gallery`）的專案名稱，系統會自動組合產生包含連字號的預設套件結構（Package Name，如 `com.echogallery.echo-gallery`）。此專案下載並匯入 IDE 後，將會噴出滿畫面的編譯期錯誤，導致程式碼完全無法編譯。

### 【底層架構與核心原理】

* **Java 識別字與語法規範（JLS）：** Java 語法嚴格規定，套件名稱（Package Name）必須符合 Java 識別字（Identifiers）的定義。在 Java 語法中，減號 `-` 是保留的**算術運算子（Subtraction Operator）**。
* **詞法分析器（Lexical Analyzer）崩潰：** 當 Java 編譯器（`javac`）對程式碼進行詞法分析時，若在套件宣告語句中遇到減號，會錯誤地將其解析為 `echo` 減去 `gallery` 的數學運算，進而引發非法的語法結構異常。
* **規格與實體的分離：** Artifact 是專案在建置工具（Maven/Gradle）與作業系統檔案系統中的唯一識別名稱（允許使用連字號）；而 Package Name 是 JVM 內部的類別加載路徑根節點，兩者遵循完全不同的命名限制。

### 【實戰解決方案與配置對應】

在按下 GENERATE 按鈕前，必須手動介入並修正網頁上的預設自動組合：

* **方案 A（直接截斷路徑 - 業界最推薦）：** * Group：`com.echogallery`
* Artifact：`echo-gallery`
* Package name 手動修正為：`com.echogallery` （直接移除後綴的連字號路徑，精簡優雅）

* **方案 B（改為底線 - 符合轉譯規範）：**
* Package name 手動修正為：`com.echogallery.echo_gallery`

## 主題三：建置工具選擇演進與 Gradle 暫存機制引發的系統磁碟空間危機

### 【問題現象與痛點】

專案源碼雖然放置於 D 槽（如 `D:\projects\echo-gallery`），但 Gradle 引擎在下載第三方依賴套件（Jar 包）時，預設會寫入 C 槽的使用者目錄中（`C:\Users\cherry\.gradle`），導致 C 槽容量迅速耗盡。若嘗試在專案根目錄的 `settings.gradle.kts` 檔案頂端直接加入 `gradle.startParameter.gradleUserHomeDir = file("D:/.gradle")`，會立刻引發 `Script compilation error: Unresolved reference 'set'` 或是屬性唯讀的編譯錯誤。

### 【底層架構與核心原理】

* **Gradle Kotlin DSL 的強型別約束：** 歷史悠久的 `Gradle - Groovy` 採用動態語言特性，允許使用 `=` 進行寬鬆的屬性補獲與賦值；而現代的 `Gradle - Kotlin` (`.kts`) 則是**靜態強型別語言**。在最新的 Gradle 核心架構中，`gradleUserHomeDir` 屬性被宣告為唯讀（ReadOnly），因此不支援 `=` 賦值，也無法調用 `.set()` 方法。
* **JVM 語法伺服器暫存機制：** 除了 Gradle 依賴快取外，VS Code 的 Java 語言伺服器（JDT Language Server）在解析大型 Spring Boot 框架時，會產生大量的「共享索引（Shared Indexes）」與語法抽象語法樹（AST）快取，這些檔案同樣預設堆積在 C 槽，必須從底層 JVM 虛擬機啟動參數進行重導向。

### 【實戰解決方案與配置對應】

捨棄在構建腳本中動態修改的錯誤路線，改由作業系統環境變數與 IDE 全域底層參數進行終極攔截：

1. **還原 `settings.gradle.kts**`：刪除所有嘗試修改 `gradleUserHomeDir` 的程式碼，使其恢復純淨狀態。
2. **作業系統環境變數動態攔截（最高優先級）**：
* 開啟系統屬性 ->「環境變數」->「系統變數」區塊中點擊新增：
* **變數名稱**：`GRADLE_USER_HOME`
* **變數值**：`D:\.gradle`


* *原理：Gradle 引擎啟動時，底層 Java 行程會優先讀取此全域環境變數，若存在則直接覆蓋內部預設的 C 槽路徑。*


3. **VS Code 全域設定檔（`settings.json`）快取搬移配置**：
按 `Ctrl + ,` 開啟設定，進入 `settings.json`，加入以下配置，將 Java 語法編譯暫存強制導向 D 槽，並優化記憶體回收（GC）效能，確保重啟時環境絕對穩定：

```json
{
    "java.sharedIndexes.location": "D:/.vscode-java-indexes",
    "java.jdt.ls.vmargs": "-XX:+UseParallelGC -XX:GCTimeRatio=4 -XX:AdaptiveSizePolicyWeight=90 -Dsun.zip.disableMemoryMapping=true -Xmx2G -Xms1G"
}

```

## 主題四：開發工具鏈（VS Code）與多版本 JDK 環境配置衝突及任務啟動異常

### 【問題現象與痛點】

當本機作業系統預設安裝的環境為 Java 17，而嘗試執行基於最新技術棧建立的 Spring Boot 專案時，主控台會拋出 `UnsupportedClassVersionError` 錯誤。與此同時，VS Code 右下角頻繁跳出核心擴充套件異常：

```text
Activating task providers java
Error: there is no registered task type 'Java'. Did you miss installing an extension that provides a corresponding task provider?

```

導致整合開發環境完全失去排錯、自動補完與 Dashboard 啟動面板的管理功能。

### 【底層架構與核心原理】

* **Java 類別檔案主版本號（Major Version）校驗機制：** JVM 具備嚴格的向前相容性，但拒絕向後相容。JDK 17 編譯出的類別檔案對應主版本號為 `61`，而最新世代的 Spring Boot 強制定界最低必須使用 JDK 21（主版本號 `65`）。當低版本的 JVM (Java 17) 嘗試去載入高版本編譯的位元組碼時，類別加載器（ClassLoader）便會主動拋出安全異常並終止行程。
* **語言伺服器協議（LSP）與任務提供者（Task Provider）斷裂：** VS Code 的 Java 擴充套件本質上是一個透過 LSP 與後台 Java 行程通訊的介面。當後台的 `java.exe` 版本與前端外掛預期設定產生嚴重衝突，或舊版本的暫存工作區（Workspace）毀損時，Java 語言伺服器行程會直接崩潰，導致前端無法註冊名為 'Java' 的自動化任務。

### 【實戰解決方案與配置對應】

1. **安裝對齊硬體架構之 JDK**：
* 下載 JDK 21 時務必明確勾選 **Architecture: x86 64-bit (x64)**（針對 Intel/AMD 晶片，切勿誤選 ARM64），確保作業系統底層系統呼叫適配。


2. **在 VS Code `settings.json` 中明確綁定 Java 執行路徑**：
排除系統環境變數混亂的干擾，直接在 IDE 內部指定高版本 SDK 的實體絕對路徑：

```json
{
    "java.jdt.ls.java.home": "C:\\Program Files\\Zulu\\zulu-21",
    "spring-boot.ls.java.home": "C:\\Program Files\\Zulu\\zulu-21"
}

```

3. **重置語言伺服器工作區暫存（SOP 大絕招）**：
按下 `Ctrl + Shift + P` 喚醒命令面板，輸入並執行 `Java: Clean Java Language Server Workspace`，在彈出的對話框中點選 `Restart and Delete`，強迫外掛清理損壞的 AST 暫存並重新載入任務提供者。

## 主題五：Spring Boot 框架內部生命週期與資料庫自動組態（Auto-Configuration）啟動崩潰

### 【問題現象與痛點】

在專案建置初期，為了長遠架構規劃而勾選了 `Spring Data JPA` 與資料庫驅動套件，但在本機尚未架設實體資料庫，或尚未在設定檔中宣告連線資訊時，點擊 Run 啟動應用程式，Spring Boot 會在列印出經典大樹標誌後瞬間崩潰，噴出長串紅色錯誤：

```text
Failed to configure a DataSource: 'url' attribute is not specified and no embedded datasource could be configured.

```

### 【底層架構與核心原理】

* **Spring Boot 的約定優於配置（Convention over Configuration）與條件註解：** Spring Boot 啟動生命週期中包含自動組態（Auto-Configuration）階段。當類別路徑下偵測到 `spring-boot-starter-data-jpa` 的 Jar 包時，框架內部的 `DataSourceAutoConfiguration` 類別會受到條件註解（Conditional Annotations）的驅使而自動激活。
* **初始化階段的條件驗證失敗：** 該自動組態類別會去尋找對應的 `DataSource` 實例。它會優先檢查 `spring.datasource.url` 等設定屬性。若發現完全沒有配置任何實體連線網址，且類別路徑下也沒有引入任何內嵌式記憶體資料庫（如 H2），條件校驗便會判定應用程式處於「無法與任何資料源通訊卻要求持久化」的矛盾狀態。為了防止系統帶病運作，框架會觸發 `FailureAnalyzers` 機制，主動拋出致命異常並強行中斷行程生命週期。

### 【實戰解決方案與配置對應】

依據專案所處的開發階段，採取不同的應變手段：

* **方案 A（配置真正的實體資料庫 - 開發與上線標準做法）：**
將專案預設的 `application.properties` 變更為層級結構更清晰、利於維運的 `application.yaml`，並填入完整的連線矩陣（以 PostgreSQL 為例）：

```yaml
spring:
  application:
    name: echo-gallery-backend
  datasource:
    url: jdbc:postgresql://localhost:5432/echogallery
    username: ${DB_USERNAME:postgres} # 優先讀取作業系統環境變數，防範金鑰外洩
    password: ${DB_PASSWORD:your_password} # 密碼安全隔離
    driver-class-name: org.postgresql.Driver
  jpa:
    database-platform: org.hibernate.dialect.PostgreSQLDialect
    hibernate:
      ddl-auto: update # 實體類別變動時自動同步結構至資料庫
    show-sql: true # 於控制台印出自動生成的 SQL 語句以供調試
    properties:
      hibernate:
        format_sql: true

```

* **方案 B（在主程式中排除自動組態 - 專注於純 API 測試階段）：**
若短期內不希望連線資料庫，可直接修改主程式入口類別（`EchoGalleryApplication.java`）上的核心註解，命令 Spring Boot 啟動時無視資料庫檢查：

```java
package com.echogallery;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration;

@SpringBootApplication(exclude = {DataSourceAutoConfiguration.class})
public class EchoGalleryApplication {
    public static void main(String[] args) {
        SpringApplication.run(EchoGalleryApplication.class, args);
    }
}

```

## 主題六：資料庫檢視器與容器化虛擬引擎隔離機制（DBeaver + Docker Volume）

### 【問題現象與痛點】

在開發全端前後端分離架構時，誤以為下載了萬用資料庫工具（DBeaver）就能直接建立並儲存資料庫數據。實質上，DBeaver 在沒有連接外部資料庫主機引擎時無法獨立運作。而在使用輕量級容器技術 Docker 部署 PostgreSQL 時，若僅僅執行標準的啟動指令，一旦 Docker 貨櫃（Container）被重啟、刪除或電腦關機，原先儲存的所有使用者測試數據與資料表將會徹底消失。

#### 為什麼不直接安裝 PostgreSQL 在本機？
* 維運成本：直接安裝會被寫入 Windows 系統服務，一旦設定搞爛，移除重灌會浪費數小時。
* 靈活度：使用 Docker 容器，若資料庫狀態弄髒了，執行 docker compose down -v 即可在一秒內銷毀重建。這是 MVP 快速迭代最需要的「後悔藥」。

### 【底層架構與核心原理】

* **DBeaver（檢視器）與 資料庫引擎（主機）的本質區別：** DBeaver 是一款基於 JDBC 協議的純圖形化界面用戶端（GUI）。它自身完全不具備任何資料庫數據儲存、交易控制（Transaction）或磁碟寫入的引擎。它僅僅扮演一個發送 SQL 指令並渲染結果集的「檢視視窗」。
* **Docker 容器文件系統的短暫性（Ephemeral Layer）：** Docker 容器利用了 Linux 核心的命名空間（Namespace）與聯合文件系統（UnionFS）來達成輕量級虛擬化隔離。容器啟動時，會在唯讀的鏡像層之上掛載一層「可寫層（Writable Layer）」。所有在容器內部資料庫寫入的數據，預設都儲存在這個臨時的可寫層中。當容器被刪除或重構時，這個可寫層會被作業系統徹底銷毀。
* **資料卷掛載（Volume Mount）的物理直通原理：** 為了打破虛擬化隔離帶來的數據流失限制，必須使用 `-v` 參數進行實體硬碟資料卷掛載。這項技術會直接修改容器核心的掛載命名空間，將宿主機的實體路徑（如 `D:/docker/postgres/data`）直接對應到容器內 PostgreSQL 儲存二進制數據的硬碟路徑（`/var/lib/postgresql/data`）。此時，資料庫引擎所有的物理寫入會直接繞過 Docker 虛擬層，透過虛擬檔案系統（VFS）直達實體硬碟，實現「軟體在虛擬膠囊內隔離執行，數據在實體 D 槽中扎根不滅」。

### 【實戰解決方案與配置對應】

透過終端機執行以下生產等級的本地容器部署指令，將資料庫主機一秒變出來，並完美保障 C 槽空間安全：

```powershell
docker run --name echo-gallery-db -e POSTGRES_PASSWORD=your_password -e POSTGRES_DB=echogallery -p 5432:5432 -v D:/docker/postgres/data:/var/lib/postgresql/data -d postgres:latest

```

**指令核心參數之維運解析：**

* `-e POSTGRES_PASSWORD=your_password`：動態注入環境變數，直接初始化資料庫系統管理員密碼。
* `-e POSTGRES_DB=echogallery`：告訴初始化腳本在啟動時自動建立一個專屬的空白資料庫。
* `-p 5432:5432`：將宿主機的 `5432` 通訊埠映射至 Docker 容器內部的 `5432` 埠口，允許外部的 DBeaver 與 Spring Boot 後端程式碼透過 `localhost:5432` 連線進來。
* `-v D:/docker/postgres/data:...`：執行實體資料卷掛載，所有測資永不丟失且完全不佔 C 槽空間。

## 主題七：個人生產維運維度評估與資料庫架構演進決策

### 【問題現象與痛點】

在規劃一個未來預計要完善上線、長期自行維運，且開放給小範圍使用者的 side-project 時，常卡在本地安裝、Docker 容器化或是雲端託管資料庫的抉擇上。若決策錯誤（如盲目在個人筆電安裝大型實體資料庫，或選擇了不具備雲端免費資源的 SQL Server），將會導致開發階段筆電記憶體（RAM）被背景服務霸佔卡死、上線部署時遭遇高昂的雲端主機授權費開銷。

### 【底層架構與核心原理】

* **傳統關聯式資料庫的資源綁定限制：** 如微軟的 SQL Server，其架構龐大且極度依賴作業系統底層常駐服務。在雲端微服務時代，SQL Server 於各主流雲端平台的免費 Tier 資源極其罕見，且上線後的商用授權費用對於獨立開發者而言是巨大的財務負擔。
* **Serverless 雲端資料庫的存算分離（Storage-and-Compute Separation）現代架構：** 當前如 Neon、Supabase 等新興雲端 PostgreSQL 託管平台，底層採用了尖端的存算分離技術。計算節點（負責解析 SQL、執行計畫）是有狀態且可隨時休眠的，而數據儲存層（負責持久化）則掛載在超大型的分散式儲存網路上。這種技術進步使得廠商能夠大方提供永久免費額度（Free Tier，如 0.5GB ~ 1GB 儲存空間），其限制的空間大小與連線次數對於少量使用者的個人產品而言，足以安穩運作數年之久。
* **環境一致性與微服務架構：** 採用雲端 Serverless 資料庫，可以讓資料庫主機直接與本地開發環境抽離。不論本地筆電如何重開機，資料庫皆 24 小時在線，且在進行後端部署練習時，程式碼不需要因為「本地變雲端」而頻繁修改設定檔網址，達成了架構的無縫遷移。

### 【實戰解決方案與配置對應】

針對追求「0元成本、無運維負擔、極致釋放本地筆電效能」的 side-project，工程師最佳化維運決策矩陣如下：

| 評估維運維度 | 路線 A：本地實體 SQL Server | 路線 B：本地 Docker 部署 PostgreSQL | 路線 C：Serverless 免費雲端資料庫（🔥 終極首選） |
| --- | --- | --- | --- |
| **本地硬碟與記憶體開銷** | 極高（常駐 Windows 背景服務，極吃 RAM） | 中等（需啟動 Docker Desktop，耗費約 1-2GB RAM） | **0 消耗**（完全由雲端廠商免費承擔，本地只需跑 IDE） |
| **雲端部署便利性** | **極差**（雲端免費資源極少，授權費昂貴） | 中等（上線部署時必須在雲端重新手動搞定一套 DB） | **極佳**（DB 本來就在雲端，後端程式部署上雲後可直連） |
| **生產環境維護開銷** | 費用高昂 | 需自行租用雲端硬碟空間並定期手動備份 | **0 元**（符合 Neon/Supabase 永久免費 Tier 規範） |
| **少量使用者承載力** | 效能過剩且架構笨重 | 依賴自身伺服器運維能力 | **完美適配**（免費 0.5G 空間足夠單純紀錄與文字使用 3~5 年） |

## 🛠️ 【工程師的應急大絕招（CLI 終極指令集）】

當整合開發環境（IDE GUI）的圖形化插件完全失靈、UI 介面卡死，或是作業系統發生底層權限異常時，以下純終端機（CLI）指令是脫離 GUI 工具、直接與核心底層溝通的終極應急解決方案。

### 1. 解決 Windows 服務控制失靈與通訊埠霸佔（處理 `Stop-Service` 權限阻斷）

* **痛點場景：** 在 PowerShell 執行 `Stop-Service -Name postgresql*` 卻無情拋出 `無法開啟 postgresql-x64-17 服務 (於電腦 '.' 上)` 錯誤。這是因為該終端機行程未取得系統管理員（Administrator）最高權限，無法向 Windows 的服務控制管理器（SCM）獲取安全控制控制代碼（Handle）。
* **純終端機強制終結方案：**
不要透過服務管理器，直接繞過去，利用網路通訊埠追蹤技術鎖定霸佔 `5432` 連線埠的底層行程 PID，並動用作業系統內建的強制毀滅訊號將其在記憶體中抹除：

```powershell
# 步驟一：精確尋找是哪一個行程（PID）霸佔了 5432 埠口
netstat -ano | findstr 5432

# 輸出範例：TCP    0.0.0.0:5432    0.0.0.0:0    LISTENING    28624
# 這裡末尾的 28624 就是行程的 PID

# 步驟二：動用系統最高強制訊號（/F），無視服務管理器的權限鎖，直接依據 PID 擊殺行程
taskkill /F /PID 28624

# 或者是直接根據映像名稱強制清除所有潛伏的 postgres 行程
taskkill /F /IM postgres.exe

```

### 2. 脫離 IDE 插件完全失靈狀態，使用純終端機進行構建與啟動

* **痛點場景：** VS Code 右下角持續噴出 `there is no registered task type 'Java'` 錯誤，或者 Spring Boot Dashboard 樹苗圖示點擊 Play 完全沒有任何反應。
* **純終端機自主構建方案：**
完全不需要依賴 IDE 插件與全域配置的 Gradle 軟體，直接利用 Spring Boot 專案根目錄下內建的構建封裝器（Gradle Wrapper）。它會自我校驗、自我下載並調用精確對齊的構建核心：

```powershell
# 步驟一：利用終端機切換至後端專案根目錄
cd D:\projects\echo-gallery

# 步驟二：調用 Gradle Wrapper 執行清理（Clean）並重新編譯，排除測試干擾（-x test）以求極速
./gradlew clean build -x test

# 步驟三：脫離 IDE 圖形面板，直接在作業系統底層行程中啟動 Spring Boot 的 Tomcat 微服務引擎
./gradlew bootRun

```
