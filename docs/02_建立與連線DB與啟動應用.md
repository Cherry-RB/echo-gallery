## 主題一：資料庫架構決策與 D 槽儲存優化

### 1. 為什麼開發 MVP 首選 Docker PostgreSQL 而非本機安裝？

* **高容錯與極速重置**：開發 MVP 初期會頻繁修改資料表結構（Schema）。如果弄爛了，Docker 只需要砍掉 Container 就能在 3 秒內換一個全新乾淨的資料庫；本機安裝一旦權限或設定搞砸，移除重裝會浪費極大時間。
* **系統乾淨度**：不需在 Windows 系統內常駐各種資料庫服務，不留垃圾。

### 2. 核心原理破除：Docker 不是隔離的嗎？為什麼資料能存在我的 D 槽？

* **觀念澄清**：Docker 容器（Container）內部確實是完美隔離的「獨立太空艙」，擁有自己獨立的 Linux 檔案系統。如果我們不介入，容器一旦關閉（`docker compose down`），裡面的測試資料就會灰飛煙滅。
* **傳送門機制（Bind Mount）**：我們透過 `docker-compose.yml` 裡的 `volumes` 設定打破了部分隔離：
```yaml
volumes:
  - ./pgdata:/var/lib/postgresql/data

```


* **右邊 (`/var/lib/postgresql/data`)**：不能改！這是容器內 PostgreSQL 固定死存放資料的 Linux 路徑。
* **左邊 (`./pgdata`)**：自由設定！這代表在宿主機（你的電腦 D 槽專案目錄下）建立一個實體資料夾。


* **結論**：資料庫**軟體與執行環境**依然完美隔離在容器內，不污染電腦；但**純數據檔案**透過傳送門死死鎖在你的 D 槽，既保護了 C 槽空間，又確保資料不會因為容器重啟而消失。

## 主題二：隱形兇手——Port 5432 佔用與連線疑難排解

### 1. 密碼明明對的，為什麼狂噴 `password authentication failed`？

* **案發主因**：Windows 本機在過去早已直接安裝過實體版的 PostgreSQL，並且註冊成 Windows 核心系統服務（如 `postgresql-x64-17`），開機就偷偷在背景啟動，**強行霸佔了 `5432` 通訊埠**。
* **衝突現象**：當你啟動 Docker 時，雖然顯示 Started，但 VS Code 的 Database Client 去連線 `localhost:5432` 時，請求直接被 Windows 本機的「老學長」攔截接走。老學長不認得你在 Docker 裡新設的 `test_password`，因此狠心拒絕連線。
* **PostgreSQL 固執特性**：PostgreSQL 只有在 `pgdata` 資料夾**完全是空的且第一次啟動時**才會寫入新密碼。因為被老學長干擾，導致初始檔案不完整，才引發後續一連串密碼錯誤。

### 2. 徹底消滅衝突的實戰指令與步驟

1. **照妖鏡指令（檢查 Windows 本機服務）**：
```powershell
Get-Service -Name *postgres*

```


2. **一勞永逸手動關閉**：
* 按下 `Win + R` 輸入 `services.msc` 開啟 Windows 服務。
* 尋找 `postgresql-x64-17`（或相關本機 Postgres 服務），點右鍵選「停止 (Stop)」。
* 再點右鍵選「內容」，將啟動類型由「自動」改為「手動 (Manual)」（防止下次開機它又爬起來搶 Port）。


3. **Docker 乾淨重置三部曲**：
```powershell
docker compose down
# 隨後至專案左側，手動將實體 pgdata 資料夾「完全刪除」
docker compose up -d

```



### 3. 名詞釐清：Database Client 裡面的 Database vs. Service

* **Service（服務 / 連線實例）**：代表「整棟商辦大樓」，即你用 Docker 跑起來的整台資料庫伺服器（`localhost:5432`），裡面包含系統權限和多個不同的保險箱。
* **Database（資料庫）**：大樓裡的「特定辦公室」（如 `echogallery` 專用保險箱），你的資料表（Table）都長在這裡面。
* *踩坑提醒*：外掛新增連線時**必須選 PostgreSQL（大象圖標）**，若誤選 MySQL（海豚），外掛會偷偷去下載 `mysql.exe` 導致兩邊語言不通。

## 主題三：VS Code Java 語言伺服器連鎖崩潰反應

### 1. 為什麼會引發 `Server process exited with code 1` 與 `no registered task type 'Java'`？

這是一場因為「優化 C 槽空間、搬移暫存」引起的連鎖車禍：

1. **外掛迷路**：在環境搬移後，VS Code 的 Java 外掛啟動時找不到你安裝的實體 JDK 21，自動降級跑去抓 C 槽外掛資料夾內自帶的「閹割精簡版 Java (Embedded JRE)」。
2. **軍師暴斃**：我們在 `settings.json` 裡塞給了它一堆優化效能的高級記憶體參數（`"java.jdt.ls.vmargs": "-XX:+UseParallelGC..."`）。精簡版 Java 吃不下這些參數，在背景**啟動瞬間直接吐血當掉 (Exited with code 1)**。
3. **主帥變成文盲**：後台軍師倒了，VS Code 瞬間失去理解 Java 的能力。當你點擊 Run 時，它就會一臉茫然地跳出警告：`there is no registered task type 'Java'`（我不知道什麼是 Java 任務）。

### 2. `settings.json` 裡的「雙胞胎 Key」衝突

在原本的設定檔中，同時出現了兩行 `java.jdt.ls.vmargs`（一行為空、一行為優化參數），這違反了標準 JSON 格式不允許重複 Key 的原則，導致外掛載入更加混亂。

### 3. 終極淨化修復方案

打開 `settings.json`（`Ctrl + Shift + P` -> 輸入 `Preferences: Open User Settings (JSON)`），將後半段徹底修正並移除重複行：

```json
    "editor.unicodeHighlight.nonBasicASCII": false,
    "editor.unicodeHighlight.ambiguousCharacters": false,
    "editor.unicodeHighlight.includeComments": false,

    // 1. 搬移語法暫存與索引，保護 C 槽空間
    "java.sharedIndexes.location": "D:/.vscode-java-indexes",

    // 2. 核心語言伺服器效能優化參數（平行垃圾回收、記憶體上限 2G）
    "java.jdt.ls.vmargs": "-XX:+UseParallelGC -XX:GCTimeRatio=4 -XX:AdaptiveSizePolicyWeight=90 -Dsun.zip.disableMemoryMapping=true -Xmx2G -Xms1G",

    // 3. 強制指定實體正牌 JDK 21 位置，斷絕外掛胡亂降級的隱患
    "java.jdt.ls.java.home": "C:\\Program Files\\Zulu\\zulu-21",
    "spring-boot.ls.java.home": "C:\\Program Files\\Zulu\\zulu-21"
}

```

## 主題四：深度解密——為何重啟 VS Code 後，未下指令專案卻能完美「通電」？

這是整個開發環境中最容易讓人產生「魔法錯錯覺」的地方。明明沒有手動去點擊 `docker compose up`，也沒有在終端機敲下啟動指令，為什麼 Spring Boot 一跑起來就能直接抓到 PostgreSQL？

這背後涉及到 **「作業系統行程隔離」** 與 **「Spring Boot 生命週期鉤子（Lifecycle Hooks）」** 的雙重底層邏輯。

### 核心原理一：三層生命週期的完全解耦（獨立王國機制）

要釐清這個現象，必須將你電腦中的組件拆解為三個**生命週期完全獨立**的層級：

1. **IDE 編輯器層（VS Code）**
* **生命週期**：當你點擊 VS Code 的 `X` 關閉或重啟時，作業系統（Windows）只會殺掉（Kill）屬於 VS Code 本身的 GUI 行程（Process）以及其附帶的 Java 擴充套件行程（Language Server）。
* **影響範圍**：它**完全無法**干涉或觸及到系統核心服務。


2. **虛擬化宿主層（Docker Desktop / WSL2）**
* **生命週期**：Docker 在 Windows 上是依賴 **WSL2（Windows Subsystem for Linux）** 建立一個獨立的微型 Linux 虛擬機核心。Docker Daemon（守護行程 `dockerd`）作為這個虛擬機的 `init`（第一號行程）在後台長駐。
* **影響範圍**：只要你沒有去點擊桌面右下角的 Docker Desktop 選擇 "Quit"，或者重啟整台電腦，這個 Linux 虛擬機就會死死地咬住系統記憶體，永不中斷。


3. **資料庫容器層（PostgreSQL Container）**
* **生命週期**：當你在前一次開發輸入了 `docker compose up -d`，其中的 `-d`（Detached mode，分離模式）是一個關鍵。它告訴 Docker Engine：「請把這個容器丟進後台，並把終端機的主控權還給我。」
* **核心真相**：一旦進入 Detached 模式，PostgreSQL 的行程就直接託管給了 Docker Daemon。**它與你的 VS Code 終端機分頁已經沒有任何父子行程的血緣關係。** > 💡 **結論**：當你關閉並重啟 VS Code 時，背景的 WSL2 虛擬機與 PostgreSQL 容器根本不知道、也不在乎這件事，它們依然在 `localhost:5432` 默默燃燒。Spring Boot 重啟後順著 TCP 埠口摸過去，自然是一秒連上。



### 核心原理二：Spring Boot 3.1+ / 4.x 的自動化生命週期鉤子

如果你的 PostgreSQL 在重啟前真的因為某種原因「沒在運作」，但 Spring Boot 啟動時它卻奇蹟似地自己活過來了，這代表你觸發了 Spring 開發團隊引入的現代自動化依賴：`spring-boot-docker-compose`。

當專案啟動時，Spring Boot 會在 **ApplicationContext 初始化階段** 悄悄執行以下硬核邏輯：

1. **偵測與尋找**：Spring Boot 會在專案根目錄下自動掃描 `docker-compose.yml` 或 `compose.yaml` 檔案。
2. **呼叫 Docker API（暗渡陳倉）**：它會透過本地的 Docker Domain Socket 向 Docker Daemon 發送請求，等同於在後台自動執行了 `docker compose start` 或 `docker compose up -d`。
3. **動態屬性注入（Dynamic Property Source）**：Spring Boot 連上容器後，會去讀取該容器對外暴露的隨機 Port、使用者名稱和密碼，然後動態覆蓋（Override）掉你 `application.yml` 裡的資料庫設定。
4. **優雅關閉（Shutdown Hook）**：當你在 VS Code 按下停止鍵關閉 Spring Boot 時，它同樣會觸發鉤子，自動去執行 `docker compose stop`（保持資料，而非 down 摧毀容器），幫你把資料庫優雅地下線。

### 🛠️ 實戰驗證：如何用工程師的方式鐵證如山？

想肉眼確認目前到底是哪一個機制在起作用，請在 VS Code 終端機執行：

```powershell
docker ps --format "table {{.Names}}\t{{.Status}}\t{{.Ports}}"

```

* 如果看見 `Up 3 hours`：代表證實了**原理一**（容器一直活在 WSL2 背景，VS Code 重啟不影響它）。
* 如果看見 `Up Less than a minute`：代表證實了**原理二**（是 Spring Boot 啟動的瞬間，動態去把 Docker 喚醒的）。

## 🛠️ 後端工程師的終極繞過大絕招

如果未來任何時候，VS Code 的 UI 介面或圖形化外掛（如 Spring Boot Boot-Dash）再次卡死、鬧脾氣、狂噴紅字，**永遠不要被編輯器綁架**。直接打開終端機，確保路徑在 `D:\projects\echo-gallery\echo-gallery-backend` 下，輸入以下純粹的指令：

```powershell
./gradlew bootRun

```

這個指令會**完全繞過 VS Code 的圖形外掛**，直接由作業系統呼叫 Gradle 引擎暴力編譯並啟動 Spring Boot。只要看到 3 秒內 Tomcat 成功在 `8080` 埠口探出頭，後端就已經完美活過來了！
