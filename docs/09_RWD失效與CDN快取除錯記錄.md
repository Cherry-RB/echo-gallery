# RWD 手機版失效問題 — 完整除錯記錄

**日期**：2026-07-04
**專案**：EchoGallery Frontend
**問題現象**：本機開發 / DevTools 手機模擬測試皆正常，但正式部署後用真實 iPhone 開啟網站，RWD 版面完全失效（桌面版三欄式版面被硬塞進手機螢幕）

---

## 一、最終結論（先講答案）

這次問題其實是 **兩個獨立 bug 疊加**在一起，缺一都不會出現我們觀察到的完整症狀：

### 🐛 Bug 1：Vite 建置工具自動使用了舊版 Safari 不支援的 CSS 語法（根本原因）

- `vite build` 正式打包時，底層的 CSS 壓縮工具（esbuild）把我們寫的標準語法
  `@media (max-width: 1200px)`
  自動優化成 CSS Media Queries Level 4 的「範圍語法」：
  `@media (width<=1200px)`
- 這個新語法要到 **Safari 16.4（2023/3 發布）** 才開始支援
- 使用者的 iPhone 13 是 **iOS 16.3.1**，剛好卡在支援門檻之前
- 結果：整個 `@media` 區塊在該裝置上被判定為「看不懂的條件」，直接被瀏覽器忽略，導致手機版 CSS 完全沒有機會套用

**解法**：在 `vite.config.ts` 加上 `build.cssTarget`，明確告訴建置工具要相容到哪個瀏覽器版本，讓它避免使用過新的語法糖。

### 🐛 Bug 2：Render / Cloudflare 的 HTML 快取策略不當（放大問題、造成排查困難的元兇）

- Render 靜態網站背後是架在 Cloudflare CDN 上
- 預設情況下 `index.html` 的 `Cache-Control` 是 `public, max-age=0, s-maxage=300`，代表 Cloudflare 邊緣節點可以快取這份 HTML **長達 5 分鐘**
- 因為不同使用者（不同裝置、不同網路）連到的 Cloudflare 節點不同，會出現「有些人看到新版、有些人看到舊版」的不一致現象，讓所有基於裝置/瀏覽器/網路環境的排查方向全部失準
- 這也是為什麼即使已經修好 Bug 1、重新部署，仍然需要等待舊快取過期才能看到效果

**解法**：在 Render Dashboard 設定自訂 Response Header，讓 `index.html` 永遠帶上 `Cache-Control: no-cache, no-store, must-revalidate`，強迫每次都跟原始伺服器要最新版本；而真正「檔名帶 hash、內容不變」的 JS/CSS 資源則維持長期快取（`public, max-age=31536000, immutable`），兩者分開處理。

### 🐛 附帶插曲：忘記取消註解

排查過程中為了做「拿掉抽屜」的排除測試，曾暫時註解掉 `<el-drawer>`，後來復原時漏改，導致修好 RWD 後一度出現「側邊欄按鈕點不出來」的假性 bug，後續已自行發現並修正。

## 二、完整排查時間軸與每一步的判斷邏輯

除錯最大的挑戰在於：**「本機正常、正式站異常」+「不同瀏覽器/網路環境測試結果一致地異常」**，這種組合會讓人很容易先入為主地懷疑錯方向。以下記錄每一步「測試了什麼」「為什麼測這個」「結果告訴我們什麼」，這個「逐步縮小可能性範圍」的思路，比記住這次的答案更值得保留。

### 步驟 1：確認最基本的嫌疑犯 — viewport meta tag

**假設**：手機版面異常最常見的原因是 `index.html` 缺少 `<meta name="viewport" content="width=device-width, initial-scale=1.0">`，導致手機瀏覽器用桌面寬度（如 980px）渲染整個頁面再縮小顯示。

**測試方式**：直接檢查 `index.html` 原始碼。

**結果**：viewport meta tag 存在且寫法正確 → **排除**這個最常見的成因，代表要往更少見的方向查。

> 💡 **教訓**：這是初學者最常踩的坑，永遠優先排除，但排除後不要停在這裡，因為視覺症狀相似的成因還有很多種。

---

### 步驟 2：排除瀏覽器「桌面版網站」模式

**假設**：iOS Safari 有「要求桌面版網站」的設定，會忽略 viewport meta tag、強制用桌面寬度渲染。這個設定是 per-網站記憶的，容易在不知不覺間被開啟。

**測試方式**：
- 檢查 Safari 網址列 "aA" 選單的顯示文字
- 用**私密瀏覽視窗**重開（private mode 不受此設定影響的話可排除）
- 換一顆完全不同的瀏覽器（Chrome for iOS）測試

**結果**：私密瀏覽、換瀏覽器都一樣異常 → **排除**。

> 💡 **教訓**：「換一個完全獨立的瀏覽器引擎，結果卻一致」是非常強的訊號，代表問題**不在瀏覽器層**，而在更底下（作業系統層、網路層）或更上面（伺服器回應的內容本身）。這一步大幅縮小了搜索範圍。

---

### 步驟 3：懷疑並排除「內容阻擋器」（Content Blocker）

**假設**：iOS 的內容阻擋 App（如 AdGuard）是作業系統層級生效的，會同時影響 Safari 與 Chrome for iOS，且不受無痕模式影響——這個特性剛好完美對應步驟 2 觀察到的「兩個瀏覽器都異常」。

**測試方式**：檢查「設定 → Safari → 擴充功能」有無安裝任何內容阻擋器。

**結果**：沒有安裝任何擴充功能 → **排除**。

> 💡 **教訓**：即使假設最後被推翻，這一步仍然是必要的，因為它的預測（兩瀏覽器一致異常、不受無痕模式影響）跟步驟 2 的觀察完全吻合，是邏輯上優先度很高的候選。**排除一個高度吻合症狀的假設，本身就是有價值的資訊**，能讓我們對後續假設更有信心。

---

### 步驟 4：確認手機的 viewport 判斷數值是否正確

**假設**：如果手機瀏覽器本身對「自己有多寬」的判斷就出錯（例如受到系統顯示設定影響），後面所有 CSS 判斷都會連帶出錯。

**測試方式**：用第三方工具（whatsmyviewport.com）直接讀出瀏覽器回報的 `window.innerWidth` 等數值。

**結果**：顯示 390px，是 iPhone 13 該有的正常數值 → **排除**「手機認錯自己的寬度」這個方向。

> 💡 **教訓**：這一步把問題範圍從「輸入端（瀏覽器怎麼判斷寬度）」正式排除，逼問題往「處理端（CSS 邏輯本身）」或「傳輸端（拿到的內容本身）」收斂。

---

### 步驟 5：懷疑版面本身把 layout viewport 撐寬（iOS WebKit 特殊行為）

**假設**：iOS WebKit 有個特殊行為——如果頁面上有元素實際寬度超出 viewport，會自動放寬 layout viewport 去遷就它，即使 viewport meta tag 設定正確也會被蓋過。懷疑左右手機版 `<el-drawer>`（transform 位移、預設關閉但仍佔位）是元兇。

**測試方式**：直接在 `MainLayout.vue` 註解掉兩個 `<el-drawer>`，重新部署後用真機測試。

**結果**：拿掉後畫面完全沒變化 → **排除**「抽屜造成撐寬」這個假設。

> 💡 **教訓**：這是本次排查中少數用「刪減法直接驗證」而非「間接觀察」的步驟，效果很好——**當某個元件很可疑但缺乏直接證據時，與其繼續猜測，不如直接拿掉重新部署測試，用結果說話**，比堆疊更多間接推論更有效率。

---

### 步驟 6：排除電信商 / 網路層轉碼代理

**假設**：部分電信商或 Wi-Fi 環境會在網路層對網頁做「轉碼壓縮」，且自行快取轉碼結果，這種代理不受瀏覽器任何設定影響。

**測試方式**：分別用 Wi-Fi、行動數據（4G）開啟同一網址比較。

**結果**：兩者結果一致 → **排除**。

> 💡 **教訓**：这一步同樣重要的是「控制變因」的思維——固定其他所有條件（同一支手機、同一個瀏覽器），只改變網路連線這一個變因，才能乾淨地驗證或排除這個假設。

---

### 步驟 7：植入自訂診斷程式碼，把黑盒子變成白盒子（關鍵轉折點）

到這一步，所有「猜測式」的假設都已經測過一輪，卻始終沒有命中。這代表**光靠外部觀察和排除法已經無法再往下逼近**，需要真正進入頁面內部、看瀏覽器實際計算出來的數值。但使用者沒有 Mac（無法用 Safari Web Inspector 遠端除錯 iPhone），也沒有 Android（無法用 chrome://inspect）。

**解法**：在 `App.vue` 暫時塞入一段診斷用的 Vue 元件，用 `onMounted` 生命週期執行 JS，直接把關鍵數值渲染成一個蓋在畫面最上層的黑底文字浮層，這樣不需要任何開發者工具，光用手機看畫面、截圖，就能拿到平常只有 DevTools 才看得到的資訊。

這個技巧總共迭代了三版，每一版都根據上一版結果決定要挖得更深：

#### 診斷 v1：先看宏觀數據
印出 `window.innerWidth`、`document.documentElement.scrollWidth`、`window.matchMedia('(max-width:1200px)').matches`，並掃描整個 DOM 找出「實際邊界超出目前 viewport」的元素。

**結果**：`matchMedia` 回報 `true`（瀏覽器認定「現在是手機寬度」），但沒有任何元素超出邊界，`scrollWidth` 跟 `innerWidth` 一致（390px，沒有橫向溢出）。

**這個結果的矛盾之處**：如果 JS 判斷「是手機」，但視覺上桌面版側邊欄卻還在——那問題不是「撐寬」造成的（步驟 5 的方向徹底排除），而是**「本該隱藏桌面版元素的那條 CSS 規則沒有生效」**。這是一次關鍵的方向轉彎。

#### 診斷 v2：鎖定具體元素的 computed style
針對 `.desktop-only`、`.mobile-header` 等具體 class，用 `getComputedStyle()` 印出它們實際的 `display` 值。

**結果**：`.desktop-only` 顯示 `display: block`（瀏覽器對 `<aside>` 標籤的預設值），`.mobile-header` 顯示 `display: none`（寫在 media query **外面**的基礎樣式）。

**這告訴我們**：寫在 media query **外面**的樣式有生效，但寫在 media query **裡面**的樣式（不管是 `.desktop-only` 該被隱藏，還是 `.mobile-header` 該被顯示）完全沒有生效——問題精準鎖定在「這個特定的 `@media` 區塊為什麼沒有被觸發」。

#### 診斷 v3：直接讀取瀏覽器實際載入的 CSSOM 內容
用 `document.styleSheets` API 遍歷所有已載入的 CSS 規則，把所有跟 `.desktop-only` 相關、以及包含 `1200` 的 `@media` 區塊原始內容印出來。

**結果**：完全找不到任何提及 `1200` 的 media 區塊，也找不到任何 `.desktop-only`／`.mobile-header` 的規則。

**這告訴我們**：問題不是「CSS 規則存在但沒被觸發」，而是**「這份 CSS 檔案裡根本沒有這條規則」**——問題從「CSS 邏輯 / 渲染」正式轉移到「建置或部署」的範疇。

> 💡 **教訓**：當外部黑盒觀察走到瓶頸，"在頁面裡自己種一段診斷碼、把結果印在畫面上給自己看" 是一個對任何裝置都通用、不需要額外工具（Mac / Android / BrowserStack）的低成本高信息量手段。而且每一版診斷都應該根據上一版的結果**收斂範圍、往更深一層挖**，而不是每次都問一樣籠統的問題。

---

### 步驟 8：比對真機與電腦拿到的 CSS 檔名是否相同

**假設**：既然真機的 CSS 檔案裡缺少關鍵規則，有兩種可能——(a) 這份 build 打包出來本身就是壞的，所有人拿到的都一樣壞；(b) 真機拿到的其實是**舊版本**的 CSS 檔案，跟電腦拿到的不是同一份。

**測試方式**：用電腦 DevTools 開同一個網址，比對 Network 分頁裡 CSS 的**確切檔名**（Vite 用內容 hash 命名，內容不同、檔名必不同）。

**結果**：一開始電腦跟手機看到的是**同一個檔名**（`index-BdpUZeJl.css`），這代表當下這兩端拿到的是同一份 build——先傾向「build 本身有問題」，而非快取問題。

> 💡 **教訓**：這一步展示了「先假設簡單的可能性」的原則，但更重要的是——這個結論後來被下一步的新證據推翻，說明**排查過程要保持結論可修正的彈性**，不要因為某一步「看似鎖定了原因」就停止蒐證。

---

### 步驟 9：直接檢查正式部署的 HTTP Response Headers（找到放大問題的第二個 bug）

**動機**：既然懷疑轉向「build 本身壞掉」，需要先確認一件更基礎的事——Render 的部署有沒有正確更新、有沒有快取問題干擾我們的判斷。

**測試方式**：用 DevTools Network 分頁檢查該次請求的 Response Headers。

**結果**：意外看到 `Cache-Control: public, max-age=0, s-maxage=300`、`Cf-Cache-Status: HIT`、`Server: cloudflare`——這揭露了 Render 背後其實架在 Cloudflare CDN 上，且 `index.html` 這個關鍵入口檔案本身就被設計成「可以被邊緣節點快取 5 分鐘」。

**這告訴我們**：**不同使用者的請求，可能會被導到 Cloudflare 不同的邊緣節點**，各節點快取的版本、快取的時間點都可能不同——這完美解釋了為何本次除錯過程中「同一個網址、不同時間點、不同裝置測試」會得到不一致的結果，也解釋了為什麼步驟 8 一度以為兩端拿到「同一份」build，但別的時間點測試可能又不同。

**解法（第一輪）**：在 Render Dashboard 設定自訂 Header，把 `/index.html` 路徑的 `Cache-Control` 改為 `no-cache, no-store, must-revalidate`，並把 `/assets/*` 設為長期快取。

> 💡 **教訓**：這是本次除錯裡「意外之喜」的一步——原本目的只是想確認部署正確性，卻意外揭露了另一個獨立存在、一直在干擾我們判斷的變因。**這提醒我們：即使某個檢查的原始動機只是想排除某個小疑慮，也應該完整看過所有回傳的資訊（尤其是 Response Headers 這種容易被忽略的細節），因為裡面常藏著意料之外的線索。**

---

### 步驟 10：發現 Header 路徑設定的細節錯誤

**現象**：設定完 Header、也等待超過建議時間後，Response Headers 卻依然顯示舊值。

**排查**：檢查當下請求的 **Request URL**，發現實際請求路徑是網站根目錄 `/`，而非我們設定規則時填寫的 `/index.html`。雖然 Render 在伺服器端會把 `/` 對應到 `index.html` 這個實體檔案，但 Header 比對規則是**照字面路徑**做比對，並不會做這層轉換。

**解法**：額外新增一條路徑為 `/` 的相同規則。

> 💡 **教訓**：設定「路徑比對規則」類的功能時，永遠要以瀏覽器實際發出的 Request URL 為準去核對，而不是以「我認為它對應到哪個實體檔案」的直覺去核對。這是一個很容易忽略、卻會讓整組設定完全失效的細節陷阱。

---

### 步驟 11：定位到真正的根本原因——CSS Media Query Range Syntax

**線索浮現的時機點**：在步驟 9 之後，某次用 DevTools 檢查已成功載入的 `MainLayout-*.css` 內容時，直接看到編譯後的原始碼寫的是：

```css
@media (width<=1200px) { ... }
```

而不是原始碼裡手寫的 `@media (max-width: 1200px)`。

**推論**：這是 CSS Media Queries **Level 4** 新增的「範圍語法」（Range Syntax），語意完全等價於舊寫法，但屬於**新版瀏覽器才支援的語法糖**。Vite 底層的 CSS 處理工具在正式打包（production build）時，出於「幫你把 CSS 壓得更精簡」的美意，自動把舊語法轉換成了這個新寫法。

**驗證**：查詢瀏覽器相容性資料，確認 **Safari 要到 16.4（2023/3 發布）才支援這個語法**，而使用者的 iPhone 13 系統版本是 **iOS 16.3.1**——僅僅一個小版本之差，完美命中「支援門檻前一刻」。

**為什麼這能解釋前面所有現象**：
- 舊版 Safari 遇到看不懂的 media 語法，不會報錯，而是直接判定「條件不成立」、整個 `@media` 區塊靜默失效——這正是步驟 7 診斷 v1 觀察到「JS 判斷成立、CSS 卻沒生效」矛盾的真正原因（JS 的 `matchMedia()` 用的是我們手寫的標準語法，不受建置工具轉換影響，永遠判斷正確；但實際套用到頁面上的 CSS 檔案語法已經被悄悄置換）
- 電腦上的桌面 Chrome、以及 DevTools 的 iPhone 模擬模式，用的都是新版瀏覽器引擎，天生就支援這個新語法，所以從頭到尾測試都正常，才會讓人一直誤以為問題出在「裝置 / 網路 / 快取」而非「CSS 語法本身」

**最終解法**：在 `vite.config.ts` 加入
```ts
build: {
  cssTarget: ['safari14'],
  target: ['es2020', 'safari14'],
}
```
明確告訴建置工具「請把相容性目標設在 Safari 14 以上」，讓它在做語法優化時自動避開 Safari 16.4 以前不支援的新語法，改用相容性更好的傳統寫法輸出。

## 三、這次除錯歷程帶給我們的方法論總結

1. **從最常見、最低成本的假設開始排除**（viewport meta tag），但排除後要有心理準備繼續往下查，不要因為「檢查過一個常見原因」就以為萬事俱備。

2. **善用「控制變因」思維去設計每一次測試**：每次只改變一個條件（瀏覽器種類、無痕模式、Wi-Fi vs 行動數據、拿掉某個元件），觀察結果是否改變，藉此一步步縮小問題所在的「層級」（瀏覽器層？作業系統層？網路層？CSS 邏輯層？建置層？部署層？）。

3. **當外部黑盒觀察遇到瓶頸，考慮把診斷邏輯直接寫進頁面本身**，用畫面呈現原本只有開發者工具才看得到的資訊——這在缺乏遠端除錯設備（如 Mac / Android）時特別實用，且成本極低。

4. **看到矛盾的數據時，不要急著否定其中一方，而要把矛盾本身當成新線索**：JS 判斷 mobile、CSS 卻沒套用 mobile 樣式，這個矛盾正是最終破案的關鍵切入點。

5. **檢查任何回應內容時，養成習慣看過完整的 Response Headers**，即使原本的檢查動機只是想確認別的事情——快取相關的資訊常常會意外洩漏影響判斷的干擾因子。

6. **正式部署環境（production build）的行為，不能完全信任跟開發環境（dev server）一致**——建置工具在正式打包時常會做各種「你沒有主動要求，但預設會幫你做」的優化（例如這次的 CSS 語法轉換），這些優化通常對新環境無害，卻可能對你需要相容的舊環境造成無聲的破壞。遇到「本機正常、正式站異常」的情況時，值得優先懷疑建置工具本身的行為差異。

7. **除錯是一個不斷提出假設、設計測試、根據結果修正假設的循環過程**，過程中沒有查到「兇手」的每一步都不是浪費——它們幫忙把可能性範圍逐步縮小，讓最後真正命中的那一步驗證變得可能。
