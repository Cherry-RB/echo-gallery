# 除錯紀錄：今日看板時區問題排查與修復

**專案**：echo-gallery（Render + Supabase 部署）
**日期**：2026-07-10 ~ 2026-07-12
**影響範圍**：TODAY 看板每日刷新機制
**嚴重程度**：中高（核心功能體感異常，使用者每日看板無法準時更新）

---

## 一、問題現象

### 使用者回報症狀

部署於 Render（後端，Region: Singapore）+ Supabase（PostgreSQL）的間隔複習型應用程式，其「今日看板（TODAY）」功能出現以下異常：

1. 台灣時間過了 `00:00` 之後，預期今日應顯示的卡片，在 Supabase 直接下 SQL 查詢卻搜尋不到。
2. 反常地，某張卡片直到 **7/10 當天 13:00** 才刷新出現，而非在 00:00 準時出現。

### 初始查詢（使用者提供，用於重現問題）

```sql
SELECT * FROM cards c
WHERE c.user_id = 2
  AND c.next_show_at <= CURRENT_DATE
  AND c.is_archived = false;
```

### 已知環境資訊（排查前蒐集）

| 項目 | 內容 |
|---|---|
| DB session timezone | `SHOW timezone;` → `UTC` |
| 後端部署 Region | Singapore（Southeast Asia） |
| 前端 | 未設定時區 |
| 欄位型別 | `next_show_at` 為 `ZonedDateTime`（對應 PostgreSQL `timestamptz`） |
| 建卡邏輯（原始） | `ZonedDateTime.now().plusDays(intervalDays)` |
| 查詢邏輯（原始） | `nextShowAt <= ZonedDateTime.now()` |

---

## 二、排查步驟與判斷過程

### 步驟 1：確認欄位型別是否為問題根源

**動作**：檢視 Entity 與資料庫欄位定義。

```java
@Column(name = "next_show_at")
private ZonedDateTime nextShowAt;
```

**判斷**：`timestamptz`（`ZonedDateTime`）本質上儲存的是「絕對時間點（instant）」，與時區無關——資料庫內部一律轉換為 UTC 儲存，讀取時才依照 session 的 timezone 設定顯示。

**結論**：型別本身沒有問題，**排除「欄位型別需要更改」的假設**，避免了不必要且高風險的 schema migration。

---

### 步驟 2：拆解使用者手動測試用的 SQL，定位第一個根因

```sql
WHERE c.next_show_at <= CURRENT_DATE
```

**動作**：對照已蒐集到的環境資訊 `SHOW timezone;` → `UTC`。

**分析**：
- `CURRENT_DATE` 的計算基準是**當前 session 的 timezone**，而非台北時區。
- Session timezone 是 `UTC`，所以 `CURRENT_DATE` 回傳的其實是 UTC 日期，並隱式轉換為 `該UTC日期 00:00:00 UTC`。
- 台北時間 `00:00` 換算成 UTC 是**前一天的 16:00**（台北為 UTC+8），也就是說，這個查詢的「今日門檻」比台北實際的午夜**晚了 8 小時**才會翻頁。

**驗證方式（邏輯推導）**：
台北 `2026-07-10 00:00` = UTC `2026-07-09 16:00`。若此刻 `CURRENT_DATE` 仍以 UTC 計算，資料庫認定的「今天」還停留在 `2026-07-09`，因此查不到 7/10 的資料，症狀一「過 00:00 查不到」得到解釋。

**第一個根因確認**：查詢端用了 session-dependent 的 `CURRENT_DATE`，而 session timezone 是 UTC，造成日界判斷偏移 8 小時。

---

### 步驟 3：追查「為何是下午 13:00 才刷新」——比對建卡邏輯與查詢邏輯

**動作**：檢視應用程式實際使用的 Java 查詢（而非使用者手動測試用的 SQL），與建卡邏輯：

```java
// 查詢
case TODAY -> cardRepository.findTodayCards(userId, ZonedDateTime.now(), pageable);

@Query("... WHERE c.nextShowAt <= :now ...")

// 建卡
.nextShowAt(ZonedDateTime.now().plusDays(intervalDays))
```

**關鍵洞察**：這裡的比較是 `nextShowAt <= now()`，屬於**精確到時分秒的絕對時刻比較**，而不是「日期」層級的比較。

**動作**：實際查詢資料庫目前儲存的值以佐證：

```sql
SELECT id, next_show_at FROM cards ORDER BY next_show_at DESC LIMIT 10;
```

結果顯示如：`2026-07-30 07:26:18.398961+00`——帶有精確到微秒的時分秒，而非乾淨的午夜時刻。

**分析**：
- 建卡當下若是台北時間 `13:26` 呼叫 `ZonedDateTime.now()`，`plusDays(10)` 只會平移「天數」，**時分秒會原封不動保留**，變成 10 天後的 `13:26`。
- 因此該卡片必須等到「未來那天的 13:26 那一刻」，`nextShowAt <= now()` 才會成立——這正是症狀二「7/10 13:00 才刷新出來」的真正原因。
- 此問題**與時區設定無關**，即使 session timezone 改成 `Asia/Taipei` 也不會修好，因為問題不是「日期基準錯」，而是「比較粒度錯」——用「精確時刻」取代了「日曆日」的語意。

**第二個根因確認**：建卡時使用 `now()` 保留了建立當下的時分秒，導致到期判斷變成「精確到分秒的等待」，而非「整天可見」。

---

### 步驟 4：確認兩個根因是疊加關係，並排除誤判方向

在此階段整理出結論：

| 根因 | 影響 | 是否與時區有關 |
|---|---|---|
| ① `CURRENT_DATE` 用了 UTC session timezone | 過台北 00:00 查不到資料（延遲 8 小時） | 是 |
| ② 建卡邏輯保留 `now()` 的時分秒 | 卡片延遲到「建立當下的時分秒」才出現（延遲數小時到近一天） | 否，是比較粒度問題 |

兩者疊加，才會產生「過午夜查不到、卻在某個奇怪的下午時間點突然刷新」的複合現象。

**重要判斷**：這一步驟避免了「頭痛醫頭」——若只解決①（例如簡單地把 session timezone 設成 Asia/Taipei），②仍會讓卡片延遲到建卡時分秒才出現，問題不會真正解決。

---

## 三、修復方案設計

### 設計原則

> 「今天」的定義必須錨定在**業務時區（台北）的日曆日**，而不是資料庫 session 的 timezone，也不是精確到秒的絕對時刻比較。

### 3.1 資料庫欄位型別：維持不變

`timestamptz` 是正確的型別選擇（絕對時刻語意，跨時區安全），**不需要修改欄位型別，既有資料不會因型別而損毀**。DB session timezone 維持 `UTC`（Render/Supabase 預設）即可，不需更動 DB 設定，時區邏輯應收斂在應用層處理。

### 3.2 應用層新增時區錨點方法

```java
// 給查詢用：排他上界（今天結束 = 明天開始）
private ZonedDateTime getStartOfTomorrowTaipei() {
    ZoneId taipei = ZoneId.of("Asia/Taipei");
    return LocalDate.now(taipei).plusDays(1).atStartOfDay(taipei);
}

// 給建卡/回流用：今天的起算基準
private ZonedDateTime getStartOfTodayTaipei() {
    ZoneId taipei = ZoneId.of("Asia/Taipei");
    return LocalDate.now(taipei).atStartOfDay(taipei);
}
```

**設計考量（曾經歷的思考反覆）**：
- 最初嘗試只用一個 `getEndOfTodayTaipei()`（回傳明天 00:00）同時給查詢與建卡使用，結果建卡時 `getEndOfTodayTaipei().plusDays(intervalDays)` 會**多算一天**（因為基準點本身已內建 `+1`，呼叫端再疊加 `intervalDays` 造成 off-by-one）。
- 修正為拆分兩個語意不同的方法：查詢用「明天 00:00」當**排他上界**；建卡/回流用「今天 00:00」當**乾淨的起算基準**，避免偏移量藏在共用方法內部造成呼叫端無感知的誤差。

### 3.3 查詢邏輯調整：`<=` 改為 `<`，比較基準改為排他上界

```java
@Query("SELECT c FROM Card c WHERE c.user.id = :userId " +
       "AND c.nextShowAt < :startOfTomorrow AND c.isArchived = false")
Page<Card> findTodayCards(@Param("userId") Long userId,
                           @Param("startOfTomorrow") ZonedDateTime startOfTomorrow,
                           Pageable pageable);
```

**判斷邏輯**：
- 使用 `<`（排他）而非 `<=`（含等）：因為所有 `next_show_at` 修正後都會精準對齊台北午夜，若用 `<=` 在邊界時刻容易誤判「明天才該出現的卡片」提早一天顯示。
- 沒有加下限（`>= 今天 00:00`）：刻意設計，讓「逾期未看」的卡片（例如系統問題或使用者長期未開啟 App 導致 `next_show_at` 停留在過去）仍持續出現在今日看板，符合間隔複習類應用的常見產品邏輯（逾期不代表消失，而是持續提醒）。

### 3.4 建卡與回流邏輯調整：統一改用當地日曆日起算

原本：

```java
.nextShowAt(ZonedDateTime.now().plusDays(intervalDays))
```

修正為：

```java
.nextShowAt(getStartOfTodayTaipei().plusDays(intervalDays))
```

同樣邏輯套用到：`createCard`（建卡）、`updateCard`（更新間隔天數）、`snoozeCard`（稍後再看）、`readCard`（已讀推進回流時間）等四處會寫入 `nextShowAt` 的地方。

**效果**：卡片一旦設定「N 天後回流」，會精準落在「N 天後那一天的台北午夜」，不再受建立當下的時分秒影響，一過午夜即可準時出現在今日看板。

## 四、既有資料修復（Data Migration）

### 4.1 為何程式碼修好還不夠

應用層邏輯調整後，**只對未來新建、新更新的卡片生效**。資料庫既有卡片的 `next_show_at` 仍帶著建立當下的時分秒（例如 `2026-07-30 07:26:18.398961+00`），不會因程式碼部署而自動修正，這些舊卡片仍會延續舊有的錯誤行為。

### 4.2 Migration SQL 設計

```sql
UPDATE cards
SET next_show_at = date_trunc('day', next_show_at AT TIME ZONE 'Asia/Taipei')
                    AT TIME ZONE 'Asia/Taipei'
WHERE is_archived = false;
```

**邏輯拆解**：
1. `next_show_at AT TIME ZONE 'Asia/Taipei'`：將 UTC 絕對時刻轉換為台北當地時間顯示。
2. `date_trunc('day', ...)`：把台北當地時間的時分秒歸零，只留下日期部分（該日 00:00）。
3. 外層再 `AT TIME ZONE 'Asia/Taipei'`：將台北當地的午夜時刻，轉換回資料庫實際儲存所需的 UTC 絕對時刻。

### 4.3 驗證方式與曾出現的誤解

**曾經的疑惑**：跑完 UPDATE 後，直接查詢 `next_show_at` 看到的值（如 `2026-07-09 16:00:00+00`）「看起來不是 00:00」，一度懷疑 SQL 沒有生效。

**釐清**：`timestamptz` 欄位永遠以 UTC 顯示原始值，台北午夜 `00:00` 換算成 UTC 本來就是前一天 `16:00`。因此看到 `16:00:00+00` 其實**正是**「台北時間 00:00」的正確 UTC 表示法，並非執行失敗。

**正確驗證方式**：查詢時額外用 `AT TIME ZONE 'Asia/Taipei'` 轉回本地時間確認：

```sql
SELECT
    id,
    next_show_at AS before_utc,
    date_trunc('day', next_show_at AT TIME ZONE 'Asia/Taipei') AT TIME ZONE 'Asia/Taipei' AS after_utc,
    (date_trunc('day', next_show_at AT TIME ZONE 'Asia/Taipei') AT TIME ZONE 'Asia/Taipei')
        AT TIME ZONE 'Asia/Taipei' AS after_taipei_local
FROM cards
WHERE is_archived = false
ORDER BY next_show_at
LIMIT 20;
```

`after_taipei_local` 欄位若全部顯示為 `xxxx-xx-xx 00:00:00`，才代表 migration 真正成功對齊到台北午夜。實際執行後逐筆比對（例如 `id=23`：`2026-07-10 05:05:11+00` → 台北時間 `07-10 13:05` → 歸零為 `07-10 00:00` → 換算回 UTC `07-09 16:00:00+00`），確認換算邏輯無誤。

### 4.4 執行過程中踩到的第二個坑：Transaction 未真正 Commit

**現象**：依序分開執行 `BEGIN;` → `UPDATE ...;` → `SELECT ...;`（驗證）→ `COMMIT;` 四個獨立的 Run 之後，重新查詢資料，發現資料**沒有被真正修改**，仍是原始帶時分秒的舊值。

**判斷**：Supabase SQL Editor 每次按下 Run，很可能各自使用獨立的 connection/session。而 PostgreSQL 的 `BEGIN` / `COMMIT` transaction 是綁定在同一個 session 上的——分開執行時，`COMMIT` 所在的 session 跟原本 `BEGIN` 開啟 transaction 的 session並非同一個，因此形同無效指令，UPDATE 從未被真正提交。

**修正做法**：
1. 若要用 transaction 包裹，必須把 `BEGIN` / `UPDATE` / `COMMIT` **放在同一次 Run（同一個查詢框）中一次送出**，確保落在同一個 session。
2. 或更簡單地，直接執行單一 `UPDATE` 敘述（不手動包 transaction），讓資料庫以 autocommit 模式立即生效，執行完立即用另一次查詢驗證結果。

最終採用單一敘述 autocommit 方式執行，驗證通過。

### 4.5 已知遺留範圍與後續待辦

- 本次 migration 條件為 `WHERE is_archived = false`，**已封存的卡片尚未修正**。若產品邏輯允許「取消封存」讓卡片復活重新進入 TODAY 看板判斷，需評估是否要對全部卡片（含已封存）一併執行修正，或於取消封存時另行處理。
- Repository 中的統計方法 `countByUserIdAndIsArchivedFalseAndNextShowAtLessThanEqual` 與 `findTodayCards` 的比較邊界（`<=` vs `<`、`now()` vs `startOfTomorrow`）目前語意不完全一致，列為後續程式碼一致性優化項目，避免統計數字與列表筆數在極端時刻對不上。

## 五、清理善後：處理刪除卡片時的外鍵約束錯誤

在整理測試資料過程中，額外遇到並排查了一個相關的資料庫錯誤，一併記錄。

### 5.1 錯誤現象

```sql
DELETE FROM cards WHERE id = 35;
```

```
ERROR: 23503: update or delete on table "cards" violates foreign key constraint
"fkp0qio4rpeidoah30tr4e6s1dm" on table "card_tags"
DETAIL: Key (id)=(35) is still referenced from table "card_tags".
```

### 5.2 判斷

`cards` 與 `tags` 為多對多關聯，中介表 `card_tags` 持有外鍵參照 `cards.id`。資料庫拒絕直接刪除仍被引用的卡片，以防止產生孤兒資料（orphaned records），此為資料庫完整性設計的正常保護機制，而非 bug。

### 5.3 解法

先清除關聯資料，再刪除主體資料，並包在同一個 transaction（單次 Run）中執行以確保原子性：

```sql
BEGIN;

DELETE FROM card_tags WHERE card_id = 35;
DELETE FROM cards WHERE id = 35;

COMMIT;
```

### 5.4 長期優化建議

若刪除卡片為常態操作，可評估將 `card_tags` 對 `cards` 的外鍵設定調整為 `ON DELETE CASCADE`，讓刪除卡片時自動連帶清除關聯標籤紀錄，減少每次手動兩段式刪除的操作成本。

## 六、根因總結表

| 序號 | 根因 | 發現方式 | 修復方式 |
|---|---|---|---|
| 1 | 手動測試 SQL 使用 `CURRENT_DATE`，受 DB session timezone（UTC）影響，日界比實際台北午夜晚 8 小時 | 比對 `SHOW timezone` 結果與台北/UTC 時差計算 | 應用層改用明確指定 `Asia/Taipei` 計算的日界，不依賴 DB session timezone |
| 2 | 建卡時使用 `ZonedDateTime.now().plusDays(N)`，保留建立當下的時分秒，導致到期判斷變成精確到分秒的等待 | 實際查詢資料庫儲存值，發現帶有精確時分秒而非乾淨午夜 | 建卡/更新/延遲/已讀邏輯統一改用「當地日曆日 00:00 起算」 |
| 3 | 查詢與建卡曾誤用同一個帶偏移量的時間錨點方法，造成 off-by-one（多算一天） | 手動推導範例日期，發現得出結果與預期差一天 | 拆分為 `getStartOfTodayTaipei()`（建卡用）與 `getStartOfTomorrowTaipei()`（查詢排他上界用）兩個語意獨立的方法 |
| 4 | Migration SQL 執行結果誤判為「未生效」 | 查詢結果顯示 UTC 原始值（如 `16:00:00+00`），誤以為不是午夜 | 改用 `AT TIME ZONE 'Asia/Taipei'` 轉回本地時間驗證，確認邏輯正確 |
| 5 | Transaction 分次執行、COMMIT 未生效 | 執行 COMMIT 後重新查詢，資料未變更 | 改為同一次 Run 內完整送出 transaction，或改用單一敘述 autocommit |
| 6 | 刪除卡片違反外鍵約束 | 資料庫回傳 `23503` 錯誤訊息 | 先刪除中介表關聯資料，再刪除主體資料 |

## 七、學習與反思

1. **時區問題的本質是「基準點」問題，而非「型別」問題**：`timestamptz` 已經是正確且與時區無關的絕對時刻儲存方式，真正容易出錯的是「應用邏輯在計算『今天』『明天』等業務日曆概念時，選錯了計算基準（DB session timezone、伺服器所在地時區、或建立當下的隨機時分秒）」。修 bug 前應先分清楚：問題是出在「儲存」還是「業務語意計算」。

2. **絕對時刻比較 vs 日曆日比較，是兩種不同粒度的語意，不能混用**：`nextShowAt <= now()` 是精確到秒的比較，適合用在「限時倒數」類場景；但「今日看板」這種業務需求，語意上是「日曆日」的比較，應該換算成明確的日界（例如「小於明天 00:00」），而非依賴精確時刻的巧合。

3. **共用方法要留意隱藏的偏移量**：把「+1 天」這種偏移邏輯封裝進共用方法時，若呼叫端又各自疊加自己的天數，很容易在無形中造成 off-by-one，且不易被察覺。將「乾淨的錨點」與「呼叫端自行決定的偏移量」分離，是較安全的設計方式。

4. **驗證資料庫操作結果時，顯示層與儲存層可能不一致，需注意時區轉換**：`timestamptz` 欄位的顯示值一律以 UTC 呈現，若沒有主動用 `AT TIME ZONE` 轉換，容易誤判操作是否生效。

5. **雲端 SQL Editor 的 session 特性需事先了解**：手動下達涉及 transaction 控制（`BEGIN`/`COMMIT`）的敘述時，需確認該工具是否保證同一個 session；若不確定，優先採用單一敘述 + autocommit 的保守做法，降低操作風險。

6. **資料庫外鍵約束報錯是保護機制，不是意外**：遇到 `23503` 這類錯誤時，應優先理解關聯結構（先查詢是哪張表、哪個欄位引用），再決定是要調整刪除順序、包在 transaction 中處理，還是從 schema 設計層面調整 cascade 規則，而非單純視為阻礙、貿然關閉約束檢查。
