## 一、簡介

Echo Gallery 是一個讓使用者自行控制內容回流節奏的個人化瀑布流網站。
> Echo Gallery 是一個個人化內容推播與複習工具，解決使用者收藏內容後不再回看的問題。使用者可以建立外部連結卡片或自建筆記卡片，設定推播頻率，讓有價值的影音、圖文、想法與筆記在未來自動回到個人瀑布流中。專案包含卡片管理、標籤分類、週期性推播、熱度紀錄與星標冷卻機制，目標是讓使用者重新掌握內容再出現的節奏，而不是被平台演算法決定注意力。

### 解決痛點

1. **收藏死亡**：使用者常在 Bilibili、Twitter/X、Facebook、文章、影片、貼文中看到好內容，會收藏、轉存、截圖、丟 Notion。但大多數內容收藏後就死了，不會回頭去看。Echo Gallery 能讓使用者再次遇見收藏內容。
2. **演算法不可控**：現在的社群平台會推播內容，但推播權在平台手上。平台關心的是停留時間，不一定是使用者真正想複習、想反覆吸收的內容。Echo Gallery 能讓使用者將推播權拿回來，讓使用者自己設定這張卡片多久出現一次。
3. **有價值內容需要反覆出現**：很多內容不是看一次就夠。例如：觀念、金句、技術文章、學習影片、閱讀筆記、心理提醒、創作靈感、職涯方向、喜歡的作品分析。它們需要週期性回來，才能真的進入生命或知識系統。Echo Gallery 提供間隔重複的影音圖文與社群內容，讓使用者可以專注於接收值得收藏的、有價值的資訊。
4. **自建筆記也能變成推播**：Echo Gallery 不只處理外部平台收藏，也處理內部輸出，使用者可以把自己的想法、閱讀筆記、日記片段、提醒、創作靈感，做成卡片，然後它會重新出現在瀑布流裡。這就變成：「我不只是被外部內容推播，我也可以被過去的自己推播。」

### 設計理念
> 我如何讓有價值的內容，不只是被收藏後沉沒，而是可以適時的重新出現在瀑布流推播，與我相遇？

## 二、使用方式

### A、建立卡片方式

#### 卡片類型與來源

1. **連結類卡片(具體的內容來源)**：像是在notion建立筆記或知識庫一樣，複製社群平台的網址，建立一張像是貼文/卡片的內容，方便使用者之後可以直接透過點選跳轉到該內容來源。如：Bilibili、YouTube、Twitter/X、Facebook、Instagram、Threads、微博、其他網站等等。
2. **筆記類卡片(自建內容的卡片)**：像是在notion建立筆記或知識庫一樣，建立屬於自己的筆記卡片。如：思考想法、閱讀筆記等等。

#### 填寫內容

卡片的標題、推薦原因、簡介、內文、類型、連結網址、封面圖片網址、分類標籤、推播頻率等等。

### B、推播方式

應用具備瀑布流推播的看板功能。有以下幾種類型的看板：
1. **今日看板**：當未封存的卡片的「下一次出現的日期」小於或等於今日日期，就會出現在今日看板。
2. **時間序**：卡片建立日期，會顯示最新建立的卡片。
3. **熱度(星標數)**：按照卡片的星標數量，由多到少地顯示。
4. **隨機**：隨機顯示任何未封存的卡片。
5. **已封存**：顯示已封存的卡片。
6. **稍後再看**：稍後再看次數累積超過 10 次的卡片。可進一步評估是否仍具收藏價值，或是否需要調整回流頻率、暫停回流、封存或刪除。
7. **查詢**：可以選擇根據以下卡片資訊進行查詢：標題文字、標籤名稱、卡片類型、卡片建立時間、卡片id。

### C、互動方式

#### 基礎互動

1. **回流**：使用者可在建立或編輯卡片時設定回流頻率，系統會依此計算卡片「下一次出現的日期」。
2. **開啟連結**：當點選連結類卡片的來源連結，將新開視窗、跳轉至來源網頁。
3. **查看/編輯卡片詳情**：點選出現在推播牆的卡片，可進入卡片詳情頁查看與編輯卡片的完整內容。

#### 進階互動

1. **星標**：星標代表喜好度與內容價值程度。同一張卡片，使用者可以重複點擊星標，星標一段時間(如：一週)後就會恢復可以點選的功能，就像bilibili的點讚功能一樣。
2. **封存**：不希望再出現在推播牆的卡片，可以選擇封存起來。
3. **稍後再看**：針對出現在推播牆的卡片，如果目前沒有打算看，但是又希望下次回流時依然可以出現在推播牆，可以點選「稍後再看」，該卡片欄位會自動更新欄位「下一次出現的日期」。

### D、側邊欄

1. **左側導覽列**：新建卡片按鈕、導覽向各種看板的菜單、個人資料、設定、登入、登出
2. **右側狀態列**：
    * 統計資訊：已建立總卡片數量、今日回流卡片數量、稍後再看次數大於 10 的卡片數
    * 標籤排行、該標籤下的卡片數量

### 技術選型前提與限制

Echo Gallery 的技術選型以「完成 MVP 核心閉環」為優先，而不是追求完整企業級架構。

目前主要考量：

1. **先完成 MVP 功能**：優先完成卡片 CRUD、瀑布流、詳情頁、回流、星標、稍後再看、封存與登入驗證。
2. **同時作為作品集與自用工具**：技術選型需要兼顧面試展示價值與長期維護成本。
3. **預留多使用者可能性**：即使初期主要自用，仍在資料模型中保留 `user` 概念，避免未來資料隔離與多裝置同步時大幅重構。
4. **成本必須可控**：第一版不直接長期部署在 AWS，先以免費或低成本方案完成上線；AWS 作為後續雲端練習與作品集加分項。
5. **避免過度設計**：MVP 暫不導入 Spring Cloud、Redis、Message Queue、Kubernetes、複雜 Batch Job 等工具。

##### TanStack Query 引入原因

一開始專案先以自定義 composable 封裝單次 API 呼叫時的 loading、error 與 execute 狀態。
但 Echo Gallery 的核心畫面是瀑布流推播牆，後續會遇到一般單次 API 封裝不容易處理的情境：

* 使用者從推播牆進入詳情頁後，返回時希望保留原本資料與位置。
* 星標、稍後再看、封存等互動需要讓畫面即時更新。
* API 失敗時需要回復原本畫面或重新取得資料。
* 卡片列表與卡片詳情需要保持同步。

因此逐步導入 TanStack Query，用來管理從後端取得的伺服器資料狀態，例如資料快取、分頁載入、資料變更操作（mutation）與互動後資料同步。

TanStack Query 不取代 Axios；Axios 仍負責 API 請求封裝，TanStack Query 則負責管理 API 資料在前端的快取與同步。

Pinia 目前暫不作為核心工具，後續若有明確的全域 UI 狀態需求，再正式導入。

### MVP 第一版暫不導入 / 暫不優先處理的項目

| 項目 | 暫不導入 / 暫不優先處理原因 |
| :------------ | :------------------------------------ |

### 1. 2026.05.08 瀑布流卡片顯示資訊與排版

#### 待釐清需求描述：
瀑布流中卡片需要顯示哪些的資訊？哪些是重點？排版要如何設計？

#### 考量釐清：
* **卡片視圖**：Echo Gallery 的定位為建立起可以跨平台回顧收藏與個人輸出筆記的連結橋接平台。圖片、影音等多媒體資源不是儲存的重點，但在瀑布流中，仍然需要 notion gallery 視圖彈性，提供使用者可以依據偏好，自行選擇以文字為主還是圖片為主的卡片顯示方式。
* **內文長度**：考量到筆記卡片類型的內文長度可能較長，但瀑布流不能顯示太長內容。參考 twitter 或 threads 這種有字數限制的貼文卡片，可以透過下一篇貼文展開敘寫，但是本產品對「卡片」定位是一篇篇各自獨立的內容收藏，彼此之間並沒有轉發或回覆這種關聯性。因此，決議卡片需要再獨立出簡介欄位，再透過詳情頁承接長內容。參考 bilibili 的影片簡介區塊。
* **卡片重點**：不管使用者是否已經看過卡片內容、對卡片有沒有印象，收藏瀑布流的卡片顯示重點，應該要放在提醒使用者為何會選擇收藏這個內容。實際的完整內文或許不用在瀑布流顯示，但是新增「簡介」欄位是需要的，可幫助使用者快速回憶內容，「收藏理由」欄位更是需要的，這是整個產品核心。

#### summary、content、reason 區分：
* summary：給瀑布流看的簡短介紹，有字數上限。
* content：完整內容 / 長筆記。給詳情頁看的完整內容。
* reason：收藏理由。提醒使用者「當初為什麼存這個」。

優先顯示順序：reason > summary > content

> 因為在 Echo Gallery，最重要的是「為什麼這張卡值得再次出現？」

#### 新建開關欄位 showContentPreview

### 2. 2026.05.20 推播牆推播卡片的方式

#### 待釐清需求描述：
* 「今日看板」是否應該只顯示當天到期的卡片？考量到可能收藏夾裡的內容不多的情境，是否需要補上未來即將回流、過去已到期的卡片？
* 卡片只要出現在看板中，是否就代表使用者已經看過、應該自動更新卡片欄位 `nextShowAt`？還是需要設計「已回顧」按鈕，讓使用者逐張點選，以用來更新 `lastShownAt` 或 `nextShowAt`，使得卡片真的具備回流功能？

#### 考量釐清：

##### 1. `nextShowAt` 的定位：回顧卡片
卡片欄位 `nextShowAt` 為卡片下一次出現的日期。如果這個欄位小於或等於當日日期 (`nextShowAt <= today`)，就會出現在「今日看板」中。「今日看板」目的是讓使用者重新看見並回顧已到期的卡片。
##### 2. 已回顧的依據/更新 nextShowAt 時機：有效互動
* 不以「出現在推播牆」作為已回顧依據，避免：使用者可能根本沒有滑到該卡片、使用者可能只是快速略過而沒有真正閱讀、曝光判斷會引入額外複雜度 (如：重新 render 可能導致重複計算)。
* 不特地設計「已回顧」按鈕：這樣會讓 Echo Gallery 變得像待辦清單或 Anki 複習系統，增加使用者操作成本，也不符合產品想要建立的「輕量重新遇見」體驗。設計上不要求使用者逐張完成回顧，而是透過自然互動判斷卡片是否仍具有價值。
* 「已回顧」判斷方式：當使用者重新與卡片建立**有效互動**時，代表使用者重新接住這張卡片，卡片可以依照原本回流頻率安排下一次出現。
* **有效互動回顧**包含：
    * **查看卡片詳情**
    * **開啟連結類卡片連結來源**
    * **點擊星標**
    * **更新卡片內容**
> 註：此處為當時對「有效互動」的初步定義；後續第 3 點已重新釐清星標、開啟來源連結與查看詳情的互動語意，實作時以第 3 點為準。
* 第一版MVP就暫時不做後端自動更新回流時間的批次排程功能。
##### 3. 今日看板不只顯示當天到期卡片 (第一版MVP可能暫時不考慮)
* 如果推播牆只顯示當日到期的卡片，當收藏內容不多或當天到期卡片不足時，畫面可能會顯得空洞；但如果直接顯示未來才到期的卡片，又會削弱 `nextShowAt` 的意義，讓「回流日期」失去控制節奏的作用。
* 採用「回流優先，補充探索」的混合規則 (MVP 第一版可暫時先不做)：
    1. 優先顯示已到期回流卡片 (越接近今天，排序越前面)。
    2. 若數量不足，補上最近新增卡片。
    3. 若仍不足，補上隨機卡片。
* 這樣可以兼顧：
    * 回流機制的產品核心。
    * 推播牆不為空的使用體驗。
    * 內容重新發現的探索感。
##### 4. 過去到期卡片的處理方式：新增卡片互動
* 過去已到期但尚未互動的卡片，會自然符合`nextShowAt <= today`，因此會持續具備進入推播牆的資格。這樣可以確保使用者不會錯過已到期內容。
* 但若使用者一直不想互動，卡片可能長期堆積在推播牆中。因此需要提供卡片出口，讓使用者能夠主動處理不想立即互動的卡片，用來避免推播牆變成過期卡片堆積場。
* 新增卡片互動功能：「暫停回流」、「封存」、「稍後再看」功能。
#### 推播牆混合推播規則(第一版MVP暫時不做)
##### 1. 推播牆顯示優先順序
後續版本可採用以下推播牆顯示優先順序：
```text
1. 到期回流卡片：nextShowAt <= today，依 nextShowAt 由近到遠排序
2. 若數量不足，補最近新增卡片
3. 若仍不足，補隨機卡片
```
排除條件：
```text
1. 已封存卡片不進入推播牆
2. 暫停回流卡片不進入推播牆
3. 未來到期卡片不作為主推播內容
```
MVP 第一版暫時不考慮做固定比例混合流，例如 60% 到期、25% 最近新增、15% 隨機。
###### 第一優先：到期回流卡片
條件：
```ts
!card.isArchived &&
card.nextShowAt !== null &&
card.nextShowAt <= today
```
排序：
```text
nextShowAt 越接近今天，越前面
```
這樣可以讓推播牆保有「今日回流」的新鮮感，而不是變成處理 backlog 的任務清單。
###### 第二優先：最近新增卡片
如果到期回流卡片數量不足，則補上最近新增的卡片。已封存與已暫停回流的卡片，不會被補進推播牆。
```ts
!card.isArchived &&
card.nextShowAt !== null
```
###### 第三優先：隨機卡片
如果到期回流卡片與最近新增卡片仍不足，則補上隨機卡片。
隨機卡片同樣需要排除：
```ts
card.isArchived === true
```
以及：
```ts
card.nextShowAt === null
```
原因是：
* 已封存卡片代表使用者不希望它出現在主要流程。
* `nextShowAt === null` 代表卡片已暫停回流，不應再被推播牆主動顯示。

#### 新增卡片出口：暫停回流、封存、稍後再看
##### 暫停回流
暫停回流代表卡片仍存在於一般看板中，只是不再進入「今日看板」的推播。
```ts
// 資料狀態：
intervalDays = null
nextShowAt = null
```
##### 封存 `isArchived`
封存不是刪除，而是將卡片從主要推播牆與一般瀏覽流程中移出，不再參與回流。可能只出現在「已封存看板」或查詢頁。當使用者不需要再看到這張卡，但仍希望保留資料，不想刪除它，即可封存它。
```ts
// 資料狀態：
isArchived = true
```
##### 稍後再看 `snoozeCount`
稍後再看代表使用者不打算在本輪回顧回流的卡片，但是未來仍想再次遇見這張卡片，便可選擇透過稍後再看功能，讓卡片直接進入到下輪回流中，也避免過去到期但使用者不想互動的卡片持續堆積。
因此，稍後再看不代表使用者已完成回顧，而是代表使用者選擇短期延後這張卡片，所以也會更新 `nextShowAt`，但不視為有效回顧。
並新建欄位 `snoozeCount` ，代表這張卡被使用者點擊「稍後再看」的累積次數。
```ts
// 資料狀態：
snoozeDays = 3 // MVP 第一版暫定 3 天後再看
nextShowAt = today + snoozeDays // 更新下次顯示日期
snoozeCount += 1 // 代表被點擊「稍後再看」的累積次數
lastSnoozedAt = today
```
當一張卡被多次稍後再看，代表它可能不是單純「現在沒時間看」，而是使用者需要重新評估：
* 這張卡是否仍有價值？
* 是否需要更新內容？
* 是否應該暫停回流？
* 是否應該封存？
* 是否應該刪除？

MVP 第一版可先在單張卡片達到門檻 (如：`snoozeCount > 10` )時顯示提醒「這張卡已被多次稍後再看，建議重新評估是否仍有收藏價值。」。後續版本可再新增「待整理」看板，集中顯示 `snoozeCount` 達到門檻的卡片。

當使用者重新與卡片建立有效互動時，代表這張卡仍具有價值，可以重置 `snoozeCount`。
```ts
snoozeCount = 0
```
封存與刪除屬於清理型行為，不一定需要重置 `snoozeCount`。保留 `snoozeCount` 反而可以作為歷史訊號，表示這張卡可能是因為多次稍後再看而被清理。
`snoozeDays`：後續可再評估是否改為依照 `intervalDays` 或 `snoozeCount` 動態調整延後天數。
#### 卡片排程更新時機
> 註：以下為當時的初步排程規則；後續第 3 點已重新釐清星標、開啟來源連結、稍後再看與查看詳情的互動語意，實作時以第 3 點的「卡片互動與欄位更新暫定表」為準。
```text
有效回顧互動：
1. 查看詳情：重置 snoozeCount，依 intervalDays 更新 nextShowAt
2. 開啟連結：重置 snoozeCount，依 intervalDays 更新 nextShowAt
3. 星標：重置 snoozeCount，更新 likeCount，依 intervalDays 更新 nextShowAt
4. 更新卡片內容：重置 snoozeCount，視為使用者已重新整理卡片，可依 intervalDays 更新 nextShowAt
延後處理互動：
1. 稍後再看：snoozeCount + 1，短期延後 nextShowAt
2. 暫停回流：intervalDays = null，nextShowAt = null
清理型互動：
1. 封存：isArchived = true
```
#### 小結
卡片依照 `nextShowAt` 取得回流資格，但不因單純曝光而自動重新排程。只有當使用者產生有效回顧互動、選擇稍後再看，或執行暫停回流 / 封存等卡片出口操作時，系統才會更新卡片的回流狀態。其中「有效回顧互動」的實際定義與欄位更新時機，已於第 3 點重新整理。
後續可透過「到期回流 + 最近新增 + 隨機補充」的推播順序，讓 Echo Gallery 維持內容流動感；MVP 第一版可先以到期回流卡片為主，待資料量與使用情境確認後，再加入最近新增與隨機補充機制。
透過「稍後再看、暫停回流、封存」等操作，使用者可以整理不想立即互動的卡片，避免推播牆變成過期內容堆積場。

### 3. 2026.06.20 卡片互動方式與更新下次顯示欄位的時機

#### 待釐清需求描述：

* 原本規劃中，只要使用者與卡片產生互動，就可能代表使用者已經重新看見或重新接觸這張卡片，因此曾經考慮將點選星標、開啟連結類卡片來源、查看卡片詳情、點選稍後再看、更新卡片內容等行為，都視為更新 `nextShowAt` 的觸發時機。
* 但是在實際設計瀑布流互動時，發現「有互動」不一定等於「完成回顧」。不同互動背後代表的使用者意圖不同，如果全部都視為完成回顧，可能會造成不符合使用者預期的體驗。
* 因此，本次需求釐清的重點，是重新定義：什麼情境下才算使用者完成本輪回顧？哪些互動只是價值確認、來源查看、延後處理或清理卡片？

#### 需求與限制：

##### 1. 使用者體驗限制

Echo Gallery 的核心體驗不是讓使用者像完成待辦事項一樣逐張打勾，而是讓有價值的卡片以輕量方式重新出現在瀑布流中，讓使用者再次遇見它。

因此，卡片不能因為任何輕微互動就立刻從瀑布流中消失。

例如：

* 使用者在瀑布流中點擊星標，可能只是因為看到這張卡片後，再次肯定它的價值。此時使用者可能還想繼續點進卡片詳情，查看完整內容、收藏理由或筆記。如果點擊星標後卡片立刻消失，會造成使用者困惑。
* 使用者點擊連結類卡片的來源連結，可能只是想先回憶原始文章、影片或貼文內容。使用者仍可能希望回到 Echo Gallery 後，再查看卡片詳情，理解當初自己收藏這張卡片的原因。如果開啟來源連結後卡片立刻消失，也會造成使用者錯愕。
* 稍後再看代表使用者現在不想處理，但未來仍想再次遇見它，因此它是延後處理，不是完成回顧。
* 封存代表使用者想把卡片移出主要流程，是清理行為，不是完成回顧。

因此，MVP 第一版不應將所有卡片互動都視為完成回顧。

##### 2. 開發設計限制

目前 Echo Gallery 的看板仍以共用瀑布流元件為基礎。不同看板雖然有不同語意，例如今日看板、時間序看板、熱度看板、隨機看板、已封存看板、稍後再看看板與查詢結果，但它們在前端仍可能共用同一套卡片顯示與互動元件。

理想上，後續可以根據使用者是從哪一種看板進入卡片詳情，決定是否要更新 `nextShowAt`。

例如：

```text
從今日看板進入卡片詳情：
代表完成本輪回顧，更新 nextShowAt。

從時間序、熱度、查詢等其他看板進入卡片詳情：
可能只是瀏覽、管理或查找，不一定代表完成今日回顧。
```

但是如果 MVP 第一版就要精準區分不同看板來源，需要額外處理：

```text
1. 看板元件需要知道目前是哪一種 boardType。
2. 卡片元件需要知道自己是從哪一種看板被點擊。
3. 前端資料快取需要區分不同看板資料。
4. readCard API 需要只在特定看板情境下觸發。
5. 後端取得卡片詳情與更新回流狀態的責任需要進一步拆分。
```

這些設計較適合放在 MVP 後續版本逐步調整。

因此，MVP 第一版先採用較簡化的規則：暫時不完整區分看板來源，但仍先釐清不同互動的語意，避免星標、來源連結、封存等行為被錯誤視為完成回顧。

##### 3. MVP 第一版的簡化原則

MVP 第一版的目標是先完成 Echo Gallery 的核心閉環：

```text
卡片依照 nextShowAt 出現在看板
→ 使用者點進卡片詳情重新接觸卡片
→ 系統依照 intervalDays 安排下一次回流
→ 卡片未來再次出現
```

因此，MVP 第一版暫定：

> 使用者從未封存看板進入卡片詳情時，先視為完成本輪回顧，並更新 `nextShowAt`。

這是第一版的簡化設計，不代表最終版本一定會維持這個判斷。後續若看板語意與使用情境更明確，再進一步調整為「只有從今日看板進入詳情才算完成本輪回顧」。

#### 考量釐清：

##### 互動語意分類

* **完成本輪回顧**：查看卡片詳情

MVP 第一版暫時不完整區分看板來源，只要使用者從未封存看板點進卡片詳情，就先視為有效回顧。後續若看板語意更清楚，再調整為只有從今日看板進入詳情才更新 `nextShowAt`。

```ts
openCount += 1
lastOpenAt = now
lastInteractionAt = now
nextShowAt = now + intervalDays
```

* **價值確認**：點擊星標

星標代表使用者再次肯定這張卡片的價值，不代表使用者已完成本輪回顧。

```ts
likeCount += 1
lastLikedAt = now
likeAvailableAt = now + cooldownDays
lastInteractionAt = now

// 不更新 nextShowAt
// 不讓卡片立刻從看板消失
```

* **來源查看**：開啟連結類卡片的來源連結

開啟來源連結代表使用者想回到原始內容查看或回憶，不一定代表完成回顧。使用者可能仍希望回到 Echo Gallery 後查看卡片詳情與收藏理由。

```ts
// MVP 第一版不更新 nextShowAt
// 不讓卡片立刻從看板消失
```

* **延後處理**：稍後再看

稍後再看代表使用者現在不想處理這張卡，但未來仍想再次遇見它，因此是延後處理，不是完成回顧。

```ts
snoozeCount += 1
lastSnoozedAt = now
nextShowAt = now + snoozeDays
```

* **清理型互動**：封存

封存代表使用者想將卡片移出主要流程，不代表完成回顧。

```ts
isArchived = true
```

* **內容整理型互動**：更新卡片內容

更新卡片內容代表使用者重新整理這張卡片，可能屬於有效互動。

```ts
updatedAt = now
lastInteractionAt = now
```

但目前卡片更新欄位中可能也包含封存狀態，因此需要區分：

```text
內容整理型更新：可能算有效互動。
封存清理型更新：不算完成回顧，應走封存邏輯。
```

是否因更新卡片內容而同步更新 `nextShowAt`，後續再依使用情境調整。

#### 卡片互動與欄位更新暫定表

> 本表為目前互動語意與欄位更新的暫定整理，主要用於輔助 MVP 開發時判斷欄位更新責任；最終欄位與 API 行為將於 MVP 第一版完成後，於「MVP 第一版邊界」章節重新整理。

| 互動     | 互動語意    | MVP 是否更新 `nextShowAt` | 主要更新欄位                                                             | 備註                                            |

| 卡片曝光   | 單純出現在看板 |                     否 | 無                                                                  | 不因曝光更新排程                                      |
| 查看卡片詳情 | 完成本輪回顧  |                     是 | `openCount`, `lastOpenAt`, `lastInteractionAt`, `nextShowAt`       | MVP 暫時不完整區分看板來源；未封存看板點詳情先視為有效回顧               |
| 點擊星標   | 價值確認    |                     否 | `likeCount`, `lastLikedAt`, `likeAvailableAt`, `lastInteractionAt` | 不讓卡片從瀑布流消失                                    |
| 開啟來源連結 | 來源查看    |                     否 | MVP 可暫不更新                                                          | 後續可評估 `sourceOpenCount`, `lastSourceOpenedAt` |
| 稍後再看   | 延後處理    |                     是 | `nextShowAt`, `snoozeCount`, `lastSnoozedAt`                       | 不算完成回顧                                        |
| 暫停回流   | 暫停主動出現  |                 清空 | `intervalDays = null`, `nextShowAt = null`                         | 後續功能，目前可列 PENDING                             |
| 封存     | 清理型互動   |                     否 | `isArchived = true`                                                | 不算完成回顧，不需依 `intervalDays` 推進                  |
| 取消封存   | 回到可用狀態  |                   待確認 | `isArchived = false`                                               | 是否恢復 `nextShowAt` 後續再定                        |
| 更新卡片內容 | 內容整理    |                   待確認 | `updatedAt`, `lastInteractionAt`                                   | 需與封存狀態更新區分                                    |
| 刪除卡片   | 移除資料    |                     否 | 刪除資料                                                               | 不算互動統計                                        |


#### 決議：

MVP 第一版針對「是否完成回顧」採用以下簡化規則：

```text
1. 不因卡片曝光而更新 nextShowAt。
2. 不因點擊星標而更新 nextShowAt。
3. 不因開啟來源連結而立即更新 nextShowAt。
4. 使用者從未封存看板進入卡片詳情，暫時視為完成本輪回顧，並更新 nextShowAt。
5. 稍後再看代表延後處理，會更新 nextShowAt，但不算完成回顧。
6. 封存代表清理卡片，不算完成回顧。
7. 更新卡片內容可以算有效互動，但需要與封存行為區分，是否更新 nextShowAt 後續再評估。
```
