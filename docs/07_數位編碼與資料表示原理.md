# 數位編碼學習筆記

當我們學習電腦底層時，最容易混淆的就是「資料的本質」與「資料的呈現方式」。

為了理解電腦如何處理資訊，我們必須區分 **「計量單位」、「書寫語言」與「翻譯意義」**。
這裡我們用 **「數位倉庫的貨運系統」** 來理解整個架構。

---

## 一、三層模型架構

---

### 1. 基礎單位層：硬體計量 (貨運箱)

這是電腦存儲空間的計量單位，僅相當於資訊的**長度**。
(方便於計算長度、掌握容量，以及進行換算)

#### **Bit (位元)**：最小的單位

* **比喻**：**「電燈開關」**。
* **定義**：電腦只能讀懂「有電」或「沒電」。我們用 `1` 代表有電，`0` 代表沒電。
* **功用**：它是所有數位資訊的原子。單獨一個 Bit 其實沒什麼意義，就像一個開關無法表達複雜的訊息。

#### **Byte (位元組) 與 Bytes (複數)**：標準規格

* **比喻**：**「標準貨運箱」**。
* **定義**：**1 Byte = 8 Bits**。
* **為什麼是 8？**：這是工程師為了方便，強行規定「8 個開關為一組」作為最基本的處理單位。
* **差別**：Byte 是單數，Bytes 是複數。就像 1 個箱子 (1 Byte) 和 5 個箱子 (5 Bytes)。
* **功用**：這是電腦存儲空間的「最小計量單位」。當我們說檔案大小時，就是在數有多少個這樣的箱子。

---

### 2. 表示層：數字的寫法 (語言)

相當於決定我們如何對資訊進行**排列**，是我們「描述」資訊的方式，進而決定了資訊將如何**顯示**。

#### **Decimal (十進位)**：人類的母語（0-9）。
  * 功用：這是我們日常溝通、計算檔案大小、定義邏輯數值的標準格式。
  * 特性：是所有轉換的中間站，電腦處理前或處理後，通常都涉及十進位轉換。
#### **Binary (二進位)**：電腦的母語、機器的語言（0, 1）。
  * 功用：電腦硬體層的唯一邏輯，代表電流的開關狀態。
  * 定義：用 0 和 1 來代表數字。例如 `1010` 其實就是 $1 \times 2^3 + 0 \times 2^2 + 1 \times 2^1 + 0 \times 2^0 = 10$（十進位的 10）。
#### **Hex (十六進位)**：工程師的速記法（0-9, A-F）。
  * 功用：將 4 個 bits 壓成 1 個符號，方便人眼進行記憶體或位元組閱讀與除錯。
  * 定義：因為 8 個 0/1 寫起來太長 (`11110000`)，人類眼睛會看花。工程師發現，每 4 個 Bit (`1111`) 可以剛好對應 1 個十六進位符號 (`F`)。
  * 比喻：**「科學縮寫」**。
  * 邏輯說明：為什麼 256 bits 要除以 4 變成 Hex？
    * 8 bits = 1 Byte。
    * 既然 4 bits 可以壓成 1 個 Hex 字元。
    * 那 1 個 Byte (8 bits) 就剛好是 2 個 Hex 字元。
    * 所以 256 bits = 32 Bytes = 64 Hex 字元（因為 $256 \div 4 = 64$）。

* *範例：數值 65*
  * 十進位：`65`
  * 二進位：`01000001`
  * 十六進位：`41` (0x41)

---

### 3. 翻譯層：數據的意義 (字典標籤)

電腦只看得懂 0 和 1，那為什麼你螢幕上會顯示 "A" 這個字？因為有一本「翻譯字典」。
這是「編碼標準」。相當於決定我們如何對資訊進行**編碼**，進而決定了一串數字符號代表什麼真實**意義**（如文字）。

#### **ASCII (美國標準資訊交換標準碼)**

* **比喻**：**「基礎英文對照表」**。
* **定義**：規定 `01000001` 這個二進位排列 / `65` 這個十進位排列，在螢幕上要顯示為字母「A」。
* **功用**：全世界共通的基礎規範。你只要用 1 Byte (8 bits) 就能表示 256 種不同的符號（字母、數字、標點符號）。

#### **EBCDIC (IBM 擴充二進位編碼十進位交換碼)**

* **比喻**：**「古老的 IBM 方言」**。
* **定義**：這是 IBM 在早期大型主機時代搞出來的另一套翻譯字典，跟 ASCII 完全不兼容。
* **應用場景**：現在幾乎沒人用，除非你在維護幾十年前的銀行舊系統。**現代專案請直接忽略它**。

#### **編碼與進位的關係**：
  * 編碼標準（ASCII）定義了「A 是 65」。
  * 進位制（十進位/十六進位）只是決定我們在螢幕上「怎麼寫出 65 這個數」。
  * 實際電腦處理的是 65 的二進位 `01000001`。

---

### 二、關鍵概念對照表 (以字母 'A' 為例)

| 觀點 | 視角 | 單位/格式 | 具體表現 | 目的 |
| :--- | :--- | :--- | :--- | :--- |
| **硬體視角** | 機器 | Binary (Bits) | `01000001` | 電壓訊號的真實狀態 |
| **工程師視角** | 縮寫 | Hex | `41` | 人類快速閱讀的縮寫 |
| **人類視角** | 意義 | ASCII (Char) | `A` | 賦予數字實際意義 |

### 為什麼這很重要？
當我們執行 `openssl rand -base64 64` 時：
1. 產生 64 個 Bytes 的二進位亂數 (抽屜一：硬體)。
2. 進行 Base64 編碼，將其轉換為文字符號 (抽屜三：賦予人類可讀的編碼)。
3. 我們在 `.env` 中看到的 88 個字元，就是這個經過轉譯後的結果 (抽屜二：書寫方式)。

因此，**不要再糾結它是哪種進位，也不要糾結它是怎麼算的**。只要記得：我們在做的，就是把「電腦的電位訊號」，層層封裝成「人類看得懂的文字格式」。

---

### 三、為什麼要有這些轉換？

**這不是在計算數學，這是在「定義資料的樣貌」。**

1. **為什麼要除以 8 變成 Bytes？**
因為 Byte 是「硬體」的存儲規格。你買硬碟、算檔案大小，必須除以 8，因為硬體架構就是這樣綁定的。
2. **為什麼要除以 4 變成 Hex？**
因為 Hex 是「軟體除錯」的語言。人類看不懂 010101，但透過每 4 位一壓，我們能一眼看出這個 Byte 裝了什麼。這純粹是為了讓人腦更好閱讀。
3. **為什麼 ASCII/Character 是 8 bits 一個字？**
因為我們在字典裡定義了：1 個英文字 = 1 Byte。這是為了讓「文字處理」與「硬體存儲」達成一致，方便計算。

### 給你的最後總結建議

你不需要強迫自己去計算這些除法，你只需要記住這個「倉庫轉譯邏輯」：

* **原始碼 (Bits/Binary)**：是倉庫裡的電路。
* **打包 (Byte)**：是倉庫裡的標準箱子。
* **檢查單 (Hex)**：是倉庫管理員（工程師）看貨單用的縮寫。
* **貨物標籤 (ASCII/UTF-8)**：是貼在箱子外面的中文說明，讓人類（你）看得懂。

當你寫 `.env` 檔案時，你就是在填寫「貨物說明標籤」。你輸入 `openssl rand -base64 64`，這台機器產生了隨機的「箱子內容物」，然後把它貼上了「Base64 標籤」讓你能把它放進文字檔。**這整套流程，就是為了讓電腦與人類能和平共處。**

## 問題釐清：為何用指令產出的金鑰是 88 個字?

```
# JWT 安全性設定：建議至少 32 Bytes (256-bit)。
# 產生指令：openssl rand -base64 64 (這會產生 64 Bytes / 512-bit 的高強度隨機金鑰)
```

這純粹是**編碼方式（Encoding）造成的數學邏輯結果**。

簡單來說：下指令的 `64` 代表的是 **「原始位元組 (Bytes)」**，而你看到的是經過 **「Base64 編碼」** 後的 **「文字長度」**。這兩者在數學上是不同的長度。

### 1. 為什麼會是 88 個字？ (數學拆解)

我們來拆解 `openssl rand -base64 64` 的編碼邏輯：

1. **原始數據**：你要求產生 `64` 個 Bytes。
2. **位元量**：每個 Byte 是 8 bits，所以 $64 \times 8 = 512$ bits。
3. **Base64 編碼邏輯**：Base64 是一種將 3 個 Bytes 轉成 4 個字元的編碼方式。
4. **計算公式**：
* $64 \div 3 = 21.33$
* Base64 必須湊滿 4 個字元為一組，所以它會無條件進位到 $22$ 組。
* $22 \times 4 = 88$ 個字元。

這就是為什麼你會看到 **88** 這個數字。這完全符合數學定義，你的金鑰非常健康。

---

### 2. 對照表：別讓「長度」混淆了你的直覺

很多開發者會搞混，是因為我們常習慣 Hex (十六進位) 的計算方式，Hex 的長度剛好是 Bytes 的兩倍，所以 $32 \text{ Bytes} = 64 \text{ Hex Characters}$，這個剛好是 64 的數字給了人一種錯覺。

| 指令 | 產出單位 (Bytes) | 編碼方式 | 最終文字長度 | 安全強度 (bits) |
| --- | --- | --- | --- | --- |
| `openssl rand -hex 32` | 32 Bytes | Hex (16進位) | **64 字元** | 256-bit |
| `openssl rand -base64 64` | **64 Bytes** | **Base64** | **88 字元** | **512-bit** |

### 3. 架構師的誠懇建議

你現在手中的這個 `88` 個字的字串：

* **安全性更高**：你實際上獲得的是 512-bit 的強度，遠高於 JWT 標準要求的 256-bit。
* **完全相容**：JWT 驗證時是會將它解碼回 Bytes 處理的，所以它是 88 個字、64 個字還是 100 個字，**對 Spring Boot 的運算效能來說毫無區別**。

**所以請放寬心：**
不要因為「數字不是 64」而感到困惑。你現在手上的這一串 88 個字的亂碼，就是一份「超額滿足資安規格」的完美金鑰。

直接把這串 88 個字的亂碼貼進去 `.env`，你的系統資安等級就已經超越絕大多數的入門專案了！這樣解釋有讓你安心一點了嗎？
