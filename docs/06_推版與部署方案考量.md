## 一、 版本控制策略：倉庫架構評估（Monorepo vs. Split Repos）

在專案初期，如何組織前端與後端的原始碼倉庫，將直接影響開發效率與未來的自動化流水線（CI/CD）設計。

### 1. 方案深度對比分析

| 特性 / 方案 | 單一倉庫 (Monorepo) | 獨立倉庫 (Split Repos) |
| --- | --- | --- |
| **結構定義** | 前端與後端代碼存放於同一個 Git 倉庫，透過子目錄隔離（如 `/echo-gallery-frontend` 與 `/echo-gallery-backend`）。 | 前端、後端各自擁有完全獨立的 Git 倉庫與版本控制歷史。 |
| **開發協同體感** | **極佳。** 允許在單一 IDE 視窗中管理全端代碼，全端重構或 API 對接時無需切換視窗。 | 一般。需頻繁切換多個 IDE 視窗或 Git 管理介面。 |
| **變更原子性 (Atomicity)** | **極高。** 當修改一個 API 欄位契約時，後端的變更與前端的對應調整可以在**同一個 Git Commit** 中被記錄。 | 低。前後端必須各自提交 Commit，歷史紀錄在時間線上是斷裂的，難以一眼看出聯動性。 |
| **CI/CD 複雜度** | 中等。自動化平台需要設定「路徑篩選器（Path Filters）」，避免改前端卻觸發後端佈署。 | 簡單。各倉庫獨立觸發，流水線（Pipeline）邏輯單純。 |

### 2. 未來 CI/CD 疑慮澄清與可逆性評估

* **CI/CD 纯淨度澄清**：許多開發者誤以為 Split Repos 的 CI/CD 比較乾淨。然而，現代 CI/CD 工具（如 GitHub Actions、Vercel、Render）皆支援強大的 `Root Directory` 與 `on.push.paths` 宣告。例如，我們可以輕易設定：*只有當 `echo-gallery-backend/` 有變動時，才觸發後端編譯*。因此，Monorepo 絕非自動化佈署的障礙。
* **架構演進的可逆性**：現階段採取 Monorepo，未來若專案規模擴大、引入多團隊協作時，你可以透過 Git 核心工具（如 `git subtree` 或 `git filter-repo`），**完美地將指定子資料夾連同它過去所有的 Commit 歷史歷史紀錄，無損地剝離成全新的獨立倉庫**。

### 📌 結論

現階段強烈建議採取 **Private（私人屬性）的 Monorepo（單一倉庫）**。對於個人 Side Project，這能最大化開發效率，並確保版本變更歷史的完整性。

## 二、 原始碼安全防線：Git 私人倉庫的「零信任」資安防護

### 1. 核心觀念：Git 歷史紀錄的不可逆陷阱

Git 的底層設計是「唯讀快照的堆疊」。**一旦將包含敏感資料（如真實密碼、API 金鑰、資料庫憑證）的檔案提交（Commit）上去，即使你在下一個 Commit 中將該檔案刪除或改寫，該明文密碼依然會永遠留存在 Git 的歷史痕跡（Git Objects）中。** 未來若專案需要公開（Public）展示，駭客可以使用自動化爬蟲工具（如 GitGardan、TruffleHog）輕易挖出你的歷史 Commit 並竊取憑證。因此，資安防範必須做在**第一次 Push 之前**，這就是「零信任（Zero Trust）」原則。

### 2. 前後端不可上傳之敏感資產與雜訊清單

* **資料庫實體數據目錄 (`pgdata/`)**：Docker Compose 映射在本地磁碟上的實體資料夾。它包含 PostgreSQL 的二進位實體數據與你的測試資料，體積巨大且具隱私性，**絕對不可上傳**。我們只管理環境藍圖，不管理數據實體。
* **本地依賴與編譯產物**：後端的 `target/`、`.gradle/`、`build/` 以及前端的 `node_modules/`、`dist/`。這些應由雲端環境建置時動態產生，上傳會造成倉庫極度肥大。
* **私密環境組態檔 (`.env`, `*.local`)**：包含真實金鑰與資料庫核心密碼的本地檔案。
* **IDE 本地設定 (`.idea/`, `.vscode/`, `*.iml`)**：包含個人電腦開發路徑、暫存快取等與程式邏輯無關的雜訊。

### 🛠️ Monorepo 根目錄 `.gitignore` 終極範本

請務必在專案的最外層根目錄（與前後端子目錄並列處）建立此檔案，確保所有雜訊與敏感資產被完美過濾：

```gitignore
# --- 1. 全域資料庫與系統日誌 ---
pgdata/
*.log
.env

# --- 2. 後端：遞迴排除所有編譯與暫存目錄 ---
**/bin/
**/build/
**/target/
**/out/
**/.gradle/
**/*.class
**/*.jar

# --- 3. 前端：遞迴排除依賴與打包物 ---
**/node_modules/
**/dist/
**/dist-ssr/
**/*.local

# --- 4. IDE 開發工具配置 (允許擴充設定) ---
.idea/
*.iml
*.iws
*.ipr
.vscode/
!.vscode/extensions.json
.DS_Store

# --- 5. 隱私與敏感檔案 ---
docs/raw/
*.pem
*.key
*.secret
*.jks
```

## 三、 環境組態管理與金鑰注入機制

### 1. 本地明文設定是否顯得「不專業」？

**不，這在業界是非常標準且正常的作法。** 在本地開發環境（Local Dev）中，使用如 `test_user`、`test_password` 這種低強度的明文密碼完全是合規的。專業工程師與初學者的分水嶺在於：**「你是否將這些本地明文，與生產環境的真實金鑰做到了徹底的代碼隔離與動態降級（Fallback）機制」**。

### 2. 大企業多環境 (Local/UT/UAT/Prod) 的精簡化落地

大型金融機構因涉及法遵、大規模跨團隊審計與零容忍的容錯率，必須切分 Local/UT/UAT/Staging/Prod。但在個人獨立專案中照搬屬於 **Over-engineering（過度設計）**。
你只需要精簡建立兩種環境配置邏輯：**開發環境（Dev）** 與 **生產環境（Prod）**。

### 3. 變數管理大腦：利用 `Docker Compose + .env` 取代自訂腳本

針對「透過某個腳本統一管理參數值，換設備不用一個個重新設定」的想法，**你的直覺極度敏銳，這正是 DevOps 的標準思維。**

在現代雲端原生生態中，你不需要自己撰寫 `.sh` 或 `.bat` 腳本。**`Docker Compose` 配合 `.env` 檔案，本身就是最完美的、宣告式的「變數管理大腦」**。

#### 實踐架構結構：

```Plaintext
echo-gallery/ (Monorepo 根目錄)
├── .gitignore               (全域 Git 防線，內含 pgdata、.env、docs/raw/)
├── .env                     (本地私密核心組態，不上傳)
├── .env.example             (設備移轉環境範本，上傳)
├── docker-compose.yml       (基礎設施藍圖：PostgreSQL 容器，上傳)
│
├── docs/                    (全端技術與架構文檔中心 - 上傳遠端)
│   ├── 01_專案建置環境與版本.md
│   ├── 02_建立與連線DB與啟動應用.md
│   ├── 03_CROS機制學習筆記.md
│   ├── 04_CROS,BatchSize,權控機制.md
│   ├── 05_推版與部署方案考量.md
│   └── ECHO_GALLERY_MVP.md  (移出前端，全端戰略定位)
│   │
│   └── raw/                 (原始對話與 Prompt 隔離區 - 穩居本機，不上傳)
│       ├── 00_建立留存文件的復用prompt.md
│       └── 01_專案建置環境與版本_原始對話.md

└── echo-gallery-frontend/  (前端 Vue 3 專案主體)
```

#### 💡 業界標準：`.env.example` 移轉機制

在 Git 倉庫中，我們會提交一個不含真實敏感資產的範本檔案（`.env.example`）。當你更換電腦或同事要加入開發時，只需執行以下流程：

1. `git clone` 專案。
2. 複製並重新命名：`cp .env.example .env`（`.env` 被 gitignore 阻擋，永遠留在本地電腦）。
3. 打開本地 `.env` 填入該設備的專屬帳密，一鍵即可啟動。

##### 根目錄 `.env.example` 範本：

```env
# Echo Gallery 環境變數範例
# 請複製此檔案為 .env 並填入實際數值，並確保它不被上傳至 Git

# 1. 資料庫連線 (給 Docker 容器用)
POSTGRES_USER=test_user
POSTGRES_PASSWORD=test_password
POSTGRES_DB=echogallery

# 2. 資料庫連線 (給 Spring Boot 後端用)
DB_USERNAME=test_user
DB_PASSWORD=test_password

# 3. JWT 安全性設定：建議至少 32 Bytes (256-bit)。
# 產生指令：openssl rand -base64 64 (這會產生 64 Bytes / 512-bit 的高強度隨機金鑰)
JWT_SECRET=生成的亂碼

# 4. 其他應用參數
SPRING_PORT=8080

```

## 四、 程式碼與配置檔的「環境無關性」落地實踐

為了達成「代碼只寫一次，換到任何環境（Render/Supabase/AWS）都不用改程式碼」的境界，我們必須在設定檔中實施 **環境變數動態注入與 Fallback（降級預設值）機制**。此外，全面將 Properties 轉型為符合現代業界公認標準、具備樹狀層級結構與高度可讀性的 **YAML 格式**。

### 1. 後端 `application.yml` 的重構

移除後端專案原來的 `src/main/resources/application.properties`，改成 yaml 格式，並利用 `${環境變數名:本地預設值}` 語法。
(使用高級巢狀防護與對齊機制，當雲端或本地 IDE 直接啟動時，皆能自動降級匹配)

```yaml
jwt:
  secret: ${JWT_SECRET}

spring:
  application:
    name: echo-gallery-backend

  # PostgreSQL 連線設定
  datasource:
    url: ${JDBC_DATABASE_URL:jdbc:postgresql://localhost:5432/echogallery}
    username: ${DB_USERNAME:${POSTGRES_USER:test_user}}
    password: ${DB_PASSWORD:${POSTGRES_PASSWORD:test_password}}
    driver-class-name: org.postgresql.Driver

  # JPA / Hibernate 開發級日誌與自動建表組態
  jpa:
    hibernate:
      ddl-auto: update
    show-sql: true
    properties:
      hibernate:
        format_sql: true
```

* **本地 IDE 直接啟動**：系統無環境變數，Spring Boot 自動啟用右側的 `localhost` 與 `test_user`。**因此，本地開發時完全不需要建立 dev 和 prod 兩套檔案輪流切換測試。**
* **雲端平台部署**：不需改動程式碼，只需在雲端後台面板設定變數（例如 `DB_USERNAME=supabase_admin`），Spring Boot 啟動時即會動態覆蓋，無縫連上雲端。

### 2. 雙重確認優化版：`docker-compose.yml` 變數化配置

這是經過雙重確認、**移除了所有廢棄註解程式碼**，且同樣採用變數化防護的最高規格版本：

```yaml
services:
  postgres:
    image: postgres:16-alpine
    container_name: echogallery-db
    ports:
      - "5432:5432"
    environment:
      # 透過 ${變數:-預設值} 語法，確保本地與生產環境變數的動態注入能力
      POSTGRES_USER: ${POSTGRES_USER:-test_user}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD:-test_password}
      POSTGRES_DB: ${POSTGRES_DB:-echogallery}
    volumes:
      #  改成 ./pgdata，代表在目前目錄（D 槽專案下）自動建立一個 pgdata 資料夾來放資料
      - ./pgdata:/var/lib/postgresql/data
  backend:
    build:
      context: ./echo-gallery-backend
      dockerfile: Dockerfile
    ports:
      - "8080:8080"
    depends_on:
      - postgres

```

## 五、 專案部署架構之階梯式演進（Evolutionary Architecture）

### 1. 釐清盲點：本地 Docker 算進入後續階段了嗎？

**尚未。這依然屬於「第一階段：本地開發環境（Local Dev）」。**
雖然你使用了 Docker 運作 PostgreSQL，但這與真正的線上生產環境（Production）有本質上的三大技術斷層：

* **網路安全與隔離**：本地全靠 `localhost` 溫室環境。線上環境你必須面對公網（Public IP）、設定伺服器防火牆（Firewall）、購買網域，並設定 **HTTPS (SSL 憑證加密)**（否則瀏覽器會直接攔截 Vue 的連線）。
* **容器化完整度**：本地的前端與後端仍依賴你筆電的運行環境（Node.js/Java SDK）啟動；真正的雲端自建部署需要將前後端各自撰寫 `Dockerfile`，實施全容器化編排。

### 2. 部署方案四階段演進藍圖

| 演進階段 | 核心架構方案 | 技術細節 | 技術痛點 / 面試效益深度評估 |
| --- | --- | --- | --- |
| **第一階段**<br>(目前狀態) | **本地開發 MVP** | 本地 Docker 跑 DB + IDE (npm/Spring) 跑前後端。 | **專注商務邏輯**：環境封閉安全，最適合高效率將畫布、無限滾動、相簿核心功能開發完整。 |
| **第二階段**<br>(MVP 上線) | **PaaS 託管雲**<br>(免費/低成本) | Vercel (前端) + Render (後端) + Supabase (雲端 DB)。 | **快速驗證，但存在面試硬傷**：免費用戶會遭遇 Render 的**休眠機制（Cold Start）**。面試官點開網頁需乾等 50 秒讓伺服器開機，UX 較差。 |
| **第三階段**<br>(自建生產) | 便宜 VPS + <br>全 Docker 編排 | 租用 Linux VPS，利用 Docker Compose 編排 Nginx + 前端容器 + 後端容器 + DB。 | **硬核工程能力展示（強烈推薦）**：消除休眠問題。面試時可完整論述「如何自行配置 Linux 防火牆、Nginx 反向代理、Certbot 自動更新 SSL 憑證」，技術含金量極高。 |
| **第四階段**<br>(雲端展示) | **AWS 進階練習版** | EC2 / ECS + RDS 部署。 | **大廠敲門磚**：非長期開啟（成本高），專門用於面試時向大型企業展示 AWS 雲端原生架構掌控能力。 |

### 📌 演進策略建議

**兩者（階段二、三）都嘗試，是成長最快的黃金路線。** 先走第二階段，你可以用最低的代價讓作品上線，親身體驗免費平台的「休眠痛點」；隨後演進到第三階段，自行租用每月約 150 元台幣的 Linux VPS，用 Docker 一鍵編排全站。

這段「發現免費平台休眠痛點 ➔ 為了提升 UX 進行架構演進 ➔ 自建 VPS 全容器化解決問題」的歷程，將成為你面試時最有力、最具備資深工程師思維的「核心故事線」。

## 六、 首次安全推版標準作業程序 (SOP)

請嚴格遵循以下順序進行專案的第一次推版，確保敏感資料絕不入庫：

### Step 1. 結構與組態檢視

確認專案結構為 Monorepo，且後端 `application.yaml` 與根目錄 `docker-compose.yml` 已全面改寫為變數化語法。

### Step 2. 建立根目錄 `.gitignore`

在專案最外層根目錄（`echo-gallery/`）下建立 `.gitignore` 檔案，並複製貼上本文第二章節提供的終極範本。

### Step 3. 本地 Git 初始化

在該根目錄打開終端機（Terminal）執行：

```bash
git init

```

### Step 4. 嚴格審查暫存狀態（最核心防線 ⚠️）

執行以下指令，檢視所有即將被 Git 準備追蹤的檔案清單：

```bash
git status

```

* **架構師查核點**：請用眼睛逐行確認畫面上紅色的預備上傳清單中，**絕對不能出現** `pgdata/`、`node_modules/`、`.env`、`target/` 等資料夾。如果出現，代表 `.gitignore` 寫法或路徑有誤，請修正直到它們完全從清單中消失，才可執行下一步。

### Step 5. 提交本地第一次快照

確認清單乾淨無誤後，將程式碼提交至本地倉庫：

```bash
git add .
git commit -m "feat: 首次提交，完成 EchoGallery 前後端基礎架構與環境組態分離"

```

### Step 6. GitHub 遠端綁定與推版

在 GitHub 建立新倉庫，**務必勾選 Private（私人倉庫）**。複製其網址並在終端機執行：

```bash
git remote add origin <你的私人倉庫網址>
git branch -M main
git push -u origin main

```

*推版成功後，請重新整理 GitHub 網頁，確認倉庫右上角帶有 🔒 鎖頭與 **Private** 標籤。*

## 七、 專家視野：邁向雲端前的三大隱性架構坑洞

在專案功能開發成熟、準備走向雲端部署之前，架構師提醒你提早將以下三點納入代碼的架構設計中：

### 1. 跨域資源共享 (CORS) 的動態化

在本地開發時，前端 `localhost:5173` 連接後端 `localhost:8080`，你可以輕易透過全域配置放行。但當進入第二、三階段部署後，前後端分別擁有不同的實體網址（例如：前端在 Vercel，後端在 Render）。後端的 CORS 放行清單**必須改為從環境變數動態讀取**，絕不可在程式碼中寫死 `*` 或本地網址，否則會引發嚴重的資安漏洞。

### 2. 雲端環境下的圖片上傳與短暫文件系統（Ephemeral File System）

目前你的專案可能將使用者上傳的相片儲存在本地磁碟路徑（如 `D:\projects\...`）。

* **痛點**：當程式部署到 Render 免費版或 Docker 容器內部時，這些路徑將不復存在。更嚴重的是，**Render 免費版容器每次重啟（或休眠復甦）都會重置檔案系統**，導致使用者上傳的圖片無預警全部消失。
* **演進方向**：在第三階段（自建 VPS）時，可利用 Docker Volume 將圖片持久化到 VPS 的硬碟中；在第四階段（AWS）時，應將上傳邏輯抽象化，改為對接 **AWS S3** 或 **Cloudinary** 等雲端物件儲存服務。

### 3. 資料庫結構版本控制 (Database Migration)

目前後端依賴 `spring.jpa.hibernate.ddl-auto=update` 讓 Hibernate 自動幫你建表。這在本地開發很方便，但在生產環境（Prod）是非常危險的作法，可能因自動變更導致線上數據意外毀損。

* **專家建議**：當專案準備進入部署階段時，應將 `ddl-auto` 改為 `validate` 或 `none`，並引入 **Flyway** 或 **Liquibase** 工具。透過撰寫 `.sql` 變更腳本（如 `V1__init_tables.sql`），讓資料庫的版本變更也跟著 Git 一起版本控制，這才是真正專業全端架構的標配。

## 九、調整過後，本地開發環境啟動專案問題排查

本章節記錄了在 Monorepo 架構重構（將 `docker-compose.yml` 移至根目錄）過程中，所面臨的各項技術卡點、底層架構衝突與排除方案。這些問題揭示了 Docker 容器化、JVM 啟動環境變數注入與 OS 行程管理的交互複雜性。

### 主題一：Docker Build 上下文與路徑映射失效

#### 【問題現象/痛點】

將 `docker-compose.yml` 從 `backend` 子目錄移至根目錄後，執行 `docker compose up --build` 時發生錯誤，系統無法找到 `Dockerfile`，導致無法建置容器。

#### 【底層架構與核心原理】

Docker Compose 的 `build.context` 是相對於 `docker-compose.yml` 檔案位置的路徑。當檔案移至根目錄，`context` 指向子目錄時，`dockerfile` 的路徑亦需同步調整，否則 Docker Daemon 會在錯誤的路徑下尋找定義檔。

#### 【實戰解決配置】

調整 `docker-compose.yml` 的建置上下文定義：

```yaml
services:
  backend:
    build:
      context: ./echo-gallery-backend
      dockerfile: Dockerfile

```

---

### 主題二：Docker Build 過程中的測試阻塞問題

#### 【問題現象/痛點】

Docker 建置過程中，Gradle 預設會執行所有單元測試。由於測試環境或依賴尚未完全配置，導致 build 失敗。

#### 【底層架構與核心原理】

Gradle 的 `build` 任務包含 `test` 生命週期階段。在容器建置環境（環境極簡且無完整 Mock 服務）中，執行測試通常不具備意義且極易失敗。

#### 【實戰解決指令】

修改 `Dockerfile`，在 build 命令中加入 `-x test` 跳過測試，並加入 `--no-daemon` 確保在容器記憶體受限環境下穩定執行：

```dockerfile
# 修改建置指令
RUN ./gradlew build -x test --no-daemon

```

---

### 主題三：8080 Port 通訊埠衝突與服務分離策略

#### 【問題現象/痛點】

同時啟動 Docker 容器與本機 Gradle Spring Boot 應用時，兩者皆試圖綁定 `8080` Port，造成啟動衝突。

#### 【底層架構與核心原理】

作業系統的網路 Socket 綁定是獨佔式的。若兩組行程皆嘗試監聽 `0.0.0.0:8080`，後啟動者將會 Bind Fail。在 MVP 開發階段，建議採取「分離策略」。

#### 【實戰解決配置】

在開發期間，若優先使用 IDE 啟動後端，應註解掉 `docker-compose.yml` 中的 `backend` 服務設定，僅保留 `postgres` 資料庫容器，降低系統資源損耗與衝突風險。

---

### 主題四：資料庫連線 URL 的解析差異

#### 【問題現象/痛點】

在開發環境中，將 DB 連線 URL 設定為 `postgre` (容器名稱) 導致連線失敗。

#### 【底層架構與核心原理】

在 Docker 內部的 Bridge Network 中，容器間透過「容器名稱」作為 DNS 溝通。然而，若你是從「宿主機（本機 Windows/Mac IDE）」連線至容器內的資料庫，必須使用映射後的 `localhost` 與 Port，而非容器內的網路名稱。

#### 【實戰解決配置】

於本地 `.env` 設定：

```env
JDBC_DATABASE_URL=jdbc:postgresql://localhost:5432/echogallery
```

---

### 主題五：VS Code Debugger 與環境變數注入

#### 【問題現象/痛點】

Spring Boot 應用啟動時，無法讀取 `.env` 中的環境變數（如 `JWT_SECRET`），導致資安組態載入失敗。

#### 【底層架構與核心原理】

VS Code 的 Java 除錯器啟動時，預設不會自動掃描專案根目錄的 `.env` 檔案。它需要明確的 Runtime Configuration 注入才能將環境變數放入 JVM 的執行環境中。

#### 【實戰解決配置】

編輯 `.vscode/launch.json`，明確加入 `envFile` 屬性：

```json
{
    "version": "0.2.0",
    "configurations": [
        {
            "type": "java",
            "name": "Launch EchoGalleryApplication",
            "request": "launch",
            "mainClass": "com.echogallery.EchoGalleryApplication",
            "projectName": "echo-gallery-backend",
            "envFile": "${workspaceFolder}/.env"
        }
    ]
}

```

---

### 主題六：資料庫持久化目錄漂移 (Data Drift)

#### 【問題現象/痛點】

資料庫重新啟動後，舊資料全部消失，回到初始狀態。

#### 【底層架構與核心原理】

Docker Volume 使用「相對路徑」映射時，其參考點為 `docker-compose.yml` 的位置。移動設定檔位置，導致 Docker 將 `./pgdata` 解析為新的實體路徑，進而認為這是全新的掛載點。

#### 【實戰解決配置】

確保 `docker-compose.yml` 中的 volume 設定固定指向根目錄的 `pgdata`，並將舊的資料夾手動搬移至此目錄。

```yaml
volumes:
  - ./pgdata:/var/lib/postgresql/data

```

---

### 主題七：Docker 環境與 Java 環境啟動生命週期

#### 【問題現象/痛點】

對於「為什麼 Spring Boot 沒啟動時 Docker 容器會自動運作」或「為什麼啟動順序混亂」感到困惑。

#### 【底層架構與核心原理】

1. **進程隔離**：Docker 容器（如 PostgreSQL）在 Detached Mode (`-d`) 下，其生命週期完全脫離終端機。即使關閉 VS Code 或重啟 IDE，容器依然在背景的 WSL2 / Docker Desktop 中執行。
2. **自動觸發**：若有使用 `spring-boot-docker-compose` 相依套件，Spring Boot 會在啟動階段主動掃描 `docker-compose.yml` 並嘗試調用 Docker API 進行服務連結，這可能會導致啟動順序的行為與預期不符。
